01

The impossible test-coverage problem

Building a programming language is easy. Keeping one bug-free over time is nearly impossible. The trap is that features don't add — they compete. Every new capability (error types, destructors, defer, multi-threading) collides with the ones already there, and the combinatorial surface of edge cases grows faster than any test suite can cover.

“There's a reason Python still has bugs 25 years later,” Vaibhav points out. “Linux has bugs. Once you have a really core system, it's really hard to have it not have bugs, because of how everything starts interacting with itself.”

Boundary ran face-first into this while building BAML. Consider a retry function: it takes a lambda, and whatever that lambda throws, retry should throw too. Change the lambda from throwing a string to throwing an int, and the compiler has to ripple that change through every caller — while defer, destructors, and threads quietly break it in ways nobody predicted.

No matter how good tests we write, no matter how much work we do — it doesn't freaking work. — Vaibhav, Boundary
02

The big idea: a loop that heals itself

If you can't write enough tests, write agents that write code instead. That's the leap Dhillan made as an intern: stop trying to anticipate every failure and instead generate failures at scale, capture them, and feed them back as fixes.

The macro concept is a closed loop. Agents produce BAML code continuously; a second layer of agents reads the transcripts, classifies what they find, files it as structured work in Linear, and yet more agents turn that work into merged pull requests. Every loop's output becomes the next loop's input.

03

Inside the discovery engine

Work enters the system as a challenge. Challenges come from two places: a cron job that generates competitive-programming problems and “port this open-source repo to BAML” tasks, and a Slack bot affectionately named Bamy that lets the team say “try this.”

From there a coding agent (Claude, driven through Cursor) attempts the task. Two modes matter:

  • Warm start — the agent is handed the BAML skill (the docs/instructions) up front.
  • Cold start — the agent is forced to find the skill from scratch, the way a real new user would. If it can't locate baml agent install --cloud, that's itself a critical bug.

Every run is recorded in full — the entire Claude transcript, the code produced, the errors hit. A separate “trophy” agent reads that log and produces a structured report of what worked, what didn't, and what was incidentally discovered.

04

Bugs vs. inefficiencies

Not every finding is created equal. The trophy agent sorts each discovery into one of two buckets, and the distinction drives where it goes next.

🐛 Issue — a real bug

A defect in the language itself: a crash, a wrong compiler error, broken semantics. Becomes engineering work and, eventually, a PR.

📚 Inefficiency — a skill gap

The language is fine; the agent just wasn't guided well. Fixed by improving skill.md, the docs agents read — not the compiler.

The de-duplication that glues this together is deceptively powerful. When the same problem surfaces twice — say, an agent reaching for string.split during both a Huffman-coding task and a reverse-words task — the second sighting isn't a new ticket. It's fresh evidence appended to the existing one. That accumulation is what turns a stream of noise into a data-driven language.

Instead of waiting for humans to report these issues after a lot of work, we just have it write the code that humans are going to write anyway. — Vaibhav, Boundary
05

Linear as the operating system

This is the part most teams skip, and it's the part that makes everything else work. Every issue, every PR, every human comment and every agent decision lives in Linear — not in a bespoke database, not in a queue in memory. Linear is the shared state machine that both humans and agents read from and write to.

A ticket moves through a small set of statuses. Humans intervene at exactly two points: shaping the issue until it's worth acting on, and merging the result. Everything between is autonomous, bounded by a maximum number of turns so a stuck loop escalates instead of spiraling.

Figure 4
The Linear state machine
stateDiagram-v2
  [*] --> NotStarted: issue created / deduped
  NotStarted --> Redraft: human flags
  Redraft --> NotStarted: agent redrafts
  NotStarted --> Approved: human approves

  AgentWrites --> PRPrep: PR opened
  PRPrep --> PRPrep: fix-agent loop (max turns)
  PRPrep --> NeedsHuman: green or stuck
  NeedsHuman --> Merged: human merges
  NeedsHuman --> Redraft: wrong direction
  Merged --> [*]
★
Why Linear specifically? It's a human-friendly state machine with ownership, labels, views and — critically — history. “You can analyze the history of your issues, see which classes get redrafted most, and build a new loop to prevent them,” Vaibhav explains. That meta-loop only exists because the primary one leaves a trail.
06

The pull-request loop

Once an issue is approved, a Cursor agent picks it up, writes the fix, and opens a pull request. From there an elegant relay begins: CodeRabbit reviews the diff, GitHub CI/CD runs the real test suite, and a chain of fresh fix-agents responds to each comment or failing check.

The team deliberately stopped having Cursor run tests in its own sandbox. Cursor sandboxes were slow; the team already pays for beefy GitHub runners. Letting GitHub own verification cut test time from ~15 minutes down to 2–4 — and spawned a cleaner separation of concerns.

07

Where humans create leverage

The system is designed around a single question: where does a human add the most value per minute? The answer is rarely “writing the code.” It's catching the one judgment call an agent can't see.

A concrete example: an agent correctly implemented array.fill, but a human noticed two things the agent missed — the parameter should be named length not n, and the body should be implemented in Rust (it's a stdlib function) rather than BAML. A one-line comment after CI went green was enough; a fresh agent made both changes.

// after CI is green, a human leaves:
// "make this a Rust function and rename n -> length"

pub fn fill(length: usize, value: T) -> Array<T> { /* Rust */ }

🤖 Agents excel at

Detail, volume, patience. Catching that length is i64 but wasm32's max allocation is usize — a thing no human would bother to fix.

🧑 Humans excel at

Taste, naming, direction. Knowing the Rust rewrite is worth it, and that n should be length for every future reader.

There's a level of detail agents are really good at, and a level of detail humans are really good at. — Vaibhav, Boundary

The arena: A/B-testing your own docs

One of the more elegant ideas: a skill arena. Instead of arguing about how the BAML documentation should be written, the team keeps multiple candidate skills (different branches in the skill repo) and runs the same challenge against all of them.

Winner is decided by data: cost, number of turns, success rate, and a friction signal — a model judging how much the implementing agent “struggled” or hill-climbed. The surprising, repeatable result: the best skill is usually the shortest one, because BAML does the heavy lifting.

09

The changelog agent & the reaper

Two background loops round out the factory. A changelog agent sweeps each nightly release, gathers its commits, and drafts a short human-readable note (with docs and a runnable example) that's itself validated through the agent pipeline before a human eyeballs it. Canary release notes stay hand-written; the nightly ones are automated.

The reaper is even simpler — and quietly profound. An agent walks the entire backlog and re-tests each old issue against the current language. Bugs that were fixed elsewhere simply disappear on their own. As Vaibhav puts it: “sometimes issues get fixed naturally without having to actually go do this.” The system heals itself in places no one was watching.

∞
Loops beget loops. Once work is tracked as data, you can measure which classes of issues get redrafted most — and then build a new loop to prevent that whole class. Compounding improvement, not a single breakthrough.
10

Two containers and a webhook

The infrastructure is almost disappointingly simple. The entire factory runs in two containers: one that spawns the agent services (just processes), and one that drives Claude. It's fully containerized, deployable from the web, mirrored to a spare MacBook as backup. Linear sends a webhook to kick off each step — there's no polling, no bespoke orchestration framework.

11

The philosophy: stack small loops

If there's one takeaway to carry away, it's this: you do not need to mastermind an entire software factory before you start. Even the redraft-then-back-to-not-started loop is, on its own, a useful thing. The factory emerges from stacking and layering small, isolated loops, each one solving a narrow problem and handing clean state to the next.

Find places where you can use agents, figure out where the right leverage points are for humans and where they are for agents, and just start building those things and plugging them into each other. Then one day you'll wake up and 80% of all of your stuff is automated. — Dex, Human Layer

Reliable automation, in other words, grows through iteration — not through one giant breakthrough. And the measure of a mature system isn't how autonomous any single agent is; it's how cleanly the loops are connected, and how much history each one leaves behind for the next loop to learn from.

12

Key takeaways

01

Generate failures, don't anticipate them

Agents writing real code surface edge cases no test suite can enumerate.

02

Classify, then route

Split every finding into a bug (language) or an inefficiency (skill). They go to different fixers.

03

Dedupe, accumulate evidence

Repeated sightings strengthen an issue rather than spam it. Noise becomes signal.

04

Linear is the OS

A human-friendly state machine with ownership and history beats a bespoke queue.

05

Cap every loop

Max turns turn a spiraling agent into an escalation to a human.

06

Let CI own truth

Don't run tests in the agent sandbox. Use the fast runners you already pay for.

07

Humans at leverage points

Spend human attention on taste and direction, never on repetitive typing.

08

Re-test old issues

A reaper agent finds bugs that fixed themselves while no one was looking.
