## Direction options (not selected for implementation)

1. **Import a previously downloaded single-file project.** The product already exports portable HTML through `GET /api/projects/:id/html`, while project creation only accepts title/model metadata. An import flow would complete the export/import pair and let users resume work on downloaded or externally authored pages; the trade-off is defining sanitization, image ingestion, and duplicate-project semantics.
2. **Route-split the project index and editor.** `apps/client/src/main.tsx` statically imports `EditorPage`, while the build emits a 1.31 MB main chunk plus many syntax-language chunks. Lazy-loading the editor would make the project list disproportionately cheaper without changing editor behavior; the trade-off is preserving the custom editor HMR identity and providing a route fallback.
3. **Define a deployable multi-process storage/run contract.** Active runs and screenshot requests are process-local while projects are file-backed. That architecture is excellent for a local single-process tool but blocks horizontal deployment; a design spike should decide whether deployment is actually a goal before introducing shared storage, leases, or queues.

## Deferred findings

- **Project-card N+1 hydration (finding 6):** high-confidence and low-risk, but the selected security/data-durability work has higher leverage. A future plan should add a preview-only response or include bounded preview HTML in the list contract; it must not hydrate messages per card.
- **Vite advisory (finding 7):** real but restricted to Windows dev-server path handling. Refresh Vite to at least 8.0.16 in a dedicated dependency commit after the selected P1 work, then rerun the full gate.
- **Route/editor bundle split (finding 8):** worthwhile after measuring cold-load impact in a browser; build size alone is not enough to choose the final chunk boundary.
- **Stale model comment (finding 9):** fix opportunistically only in a dedicated docs-only commit; it does not justify coupling to the selected plans.
- **Generated preview uses `allow-same-origin` with `allow-scripts`:** this gives model-generated code same-origin capability, but the current documented client contract depends on parent DOM access for morphing, diagnostics, element selection, and screenshot behavior. Do not remove one sandbox token as a quick fix; first write a dedicated isolation design/spike that moves trusted preview control across an explicit boundary.

## Findings considered and rejected

- **Transitive Hono advisory `GHSA-88fw-hqm2-52qc`:** Hono is present through MCP SDK dependencies, but the audited custom Node HTTP server does not import or configure Hono's CORS middleware. Reachability was not demonstrated, so it is not presented as a confirmed application vulnerability; re-check on Mastra/MCP upgrades.
- **Refactor `apps/server/src/mastra/route.ts` solely because Fallow ranks it as the top hotspot:** not worth doing now. It is highly churned and large, but has extensive characterization coverage (`route.test.ts` is 2,609 lines and route line coverage was 87.84%); refactor only around a concrete behavior change.
- **Deduplicate the 5.75% clone report immediately:** most surfaced clones are test setup in `route.test.ts`. Extracting a large test DSL now risks making scenarios less readable without a production behavior or gate improvement.
- **Add tests for every Fallow static coverage gap:** rejected as metric-driven work. The high-risk server path already enforces coverage, while many reported gaps are package config entry points and generated UI primitives.
