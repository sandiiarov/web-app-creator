# Mastra Agent DOX

## Purpose

- Owns Mastra landing-page agent impl + custom server-side streaming protocol.

## Ownership

- `agents/`: landing-page agent factory + singleton reg config; system prompt has hashline `read`/`find`/`edit` quick-ref + working guidelines. No `design` skill, no planner — agent implements the user's request directly.
- `tools/`: Mastra tool factories + landing tool registry — scrape/read/find/edit/generate_image/screenshot; legacy grep helpers stay internal, not public landing tools.
- `lib/`: OpenRouter model config, OpenRouter vision OCR (`image-ocr.ts`), bounded fetch helper (`bounded-fetch.ts`), Cloudflare Browser Run screenshot capture (`project-screenshot.ts`) via remote CDP + `playwright-core` (no local browser), cost accounting, anchored HTML doc/edit/find/image/SSE helpers, file-backed project storage (`project-store.ts`) — append-only debug logs (`client-messages.jsonl`, `agent-messages.jsonl`, `vision-messages.json`), screenshot byte persistence, hydration via shared `@workspace/conversation` reducer (`replayClientEvents`), `readAgentRawByTurn` (last per-turn Mastra snapshot for replay).
- `route.ts` claims one process-local active run per `projectId` right after project lookup, rejects overlap w/o mutating metadata or appending 2nd prompt. Records one inbound prompt (`dir:"in"`, `type:"prompt"`) w/ supplied client `turnId` (generated fallback only when omitted), maps Mastra `fullStream` chunks through `emit(event,payload)`: each outbound event queued to `client-messages.jsonl` before best-effort SSE delivery — destroyed/throwing socket can't prevent replay. Prompt attachment OCR gets same run abort signal as Mastra gen — incl image downloads, provider fetch, retry delay. Successful content-changing edits emit `html_update`; model retries emit `retry`; tool errors/malformed edit results emit terminal error tool states. Normal edit failures stay recoverable until `MAX_EDIT_FAILURES`; optional cost cap stays fatal. Cost/stats accounting emits rolling snapshots after provider cost metadata, every completed LLM step, attachment analysis, each terminal tool event; final snapshot still runs in `finally` for success, stream throws, aborts — provider-reported values only. Non-fatal aborts retain `finishReason:"stopped"` even if Mastra reports `tripwire`. Accepted runs persist terminal `stats`, optional `error`/stopped, then `done`; unconditional outer cleanup conditionally releases registry owner, flushes logs, best-effort ends response regardless of socket writes. `stopLandingAgent(projectId)` aborts attachment/Mastra/screenshot-capture w/o closing SSE. Client SSE/browser disconnect = delivery loss, not cancellation — accepted runs continue in same server process, persist terminal events regardless. Agent messages snapshotted per step + run end, generated images persisted at tool-result time, unchanged draft still gets empty-draft terminal error.
- Attachment vision OCR passes active run signal into `boundedFetch`; composes external cancellation w/ each per-attempt timeout, cancels retry sleep, propagates external abort immediately while preserving existing 5xx/internal-timeout retries.
- `index.ts`: Mastra instance, storage, logger, observability setup.

## Local Contracts

- Agent edits project HTML via hashline edit engine (first-party, vendored under `lib/hashline/`): `read`/`find` emit `[#TAG]` snapshot header + `N:TEXT` rows, record content-hash snapshot; `edit` takes `{ action, diff }` (hashline DSL), verifies `#TAG` vs snapshot (rejects stale/fabricated tags w/ re-read instruction), runs HTML-tag-balance guard (conservative adjacent-duplicate autofix via `lib/hashline/autofix.ts`, proceeds w/ warning when rebalances), then writes. Store = write-through `createProjectHtmlStore` over `html.json` (persisted source of truth); tools never mutate repo files, UI never writes HTML. `index.html` = legacy import-only. Initial/fallback reads use `getProject`; live edits stream rendered `html_update` SSE payloads (not persisted into turns). `lib/hashline/` core exempt from project's perfectionist sort + `no-explicit-any` style rules (`apps/server/oxlint.config.ts` override); unused vendored surface deleted down to what consumer uses, so `fallow:dead-code` passes w/ no exclude. First-party, no external attribution.
- Project title set server-side from first prompt (`setTitleIfUntitled`) on `POST /agent`; `ProjectMeta` persists all three per-role models (`textModel`/`imageModel`/`visionModel`) via `PATCH /api/projects/:id`, while `POST /agent` run-start persists only `textModel`, each saved message turn keeps text model used for that turn. All `project.json` read-modify-writes synchronous (`readMetaSync`/`writeMetaSync`) so model PATCH serializes vs edit-time `markHasHtmlSync` writes w/ no lost update (no `hasHtml` reversion).
- Project conversation history server-owned, persisted as append-only logs, each holds one execution context, appended per event/step (never overwritten) so data at any moment inspectable: (1) `client-messages.jsonl` — client wire (`dir:"out"` = each SSE event; `dir:"in"` = prompt + screenshot POST-back, metadata + file pointer only, never base64); `getProject` `replayClientEvents`-reduces into `ProjectMessageTurn[]` for hydration (each `dir:"in"` prompt starts turn w/ real `turnId`; `done` terminalizes; tools left running terminalized to `error` on restore). (2) `agent-messages.jsonl` — per-step Mastra `messageList` snapshots (`stream.messageList.get.response.db()` via `onStepFinish` + once at run end), `reasoning` parts stripped (`stripReasoning`) so replay doesn't inflate next prompt; `readAgentRawByTurn` returns last snapshot per turn. Both fall back to legacy `messages.json`/`raw-messages.json` for old projects. `streamLandingAgent` replays agent snapshots verbatim into next OpenRouter agent call so model sees previous tool calls/results instead of lossy prose paraphrase; falls back to prose only when turn has no agent messages. (3) `vision-messages.json` — per OCR/vision call (attachment `ocrImageInputs` AND scrape `ocrImages`, both carry `projectId`/`turnId` threaded through `LandingToolContext`), text/usage/cost only. (4) `screenshots/<NNN>-<requestId>.<ext>` — captured screenshot bytes written at POST-back resolve. Prompt image attachments OCR analyzed before OpenRouter agent run; selected-element attachments also pass captured `outerHTML` into agent context alongside screenshot OCR. No base64 image bytes ever written to JSON log.
- User-facing agent text stays concise, uses latest user prompt's language (English when English or ambiguous), must NOT echo internal tool transcripts like `Tool read done`, `Action:`, `Detail:`, or `Result:`; UI renders tool status separately.
- `LANDING_AGENT_INSTRUCTIONS` (`agents/landing-page-agent.ts`) includes advisory `HASHLINE_SYSTEM_GUIDANCE` quick-ref from `lib/hashline/edit-prompt.ts`: snapshot headers/rows, `SWAP`/`DEL`/`INS`, `+` insertion-row encoding (incl CSS `--variables`), balanced closing tags, fresh tags, failure recovery. Batches complete logical change (whole section, related block of edits, entire fix/refinement) into single medium-sized `edit` whose `diff` may carry many `SWAP`/`DEL`/`INS` ops (and several `[#TAG]` sections), instead of many tiny one-line edits; full finished page in one edit still avoided (`maxOutputTokens` truncation, `finishReason=length`). `screenshot` = final verification/QA step taken once page or requested change complete (like tests/lint), not after every edit; targeted request may screenshot once before to assess current state, once after — never reflex after each intermediate edit.
- Every user-visible tool call must include `action`; client renders it in conversation UI. Tools accepting `action` should receive one from agent. `design` skill (and its `skill`/`skill_read`/`skill_search` tools) removed; route's `skill*` display branches stay as harmless dead code.
- Mastra `tool-result` chunks w/ top-level `{ ok: false }` = terminal failures for custom SSE protocol, must stream + record `tool_call.state = "error"` even when SDK chunk `isError` flag false.
- `tools/landing-tools.ts` = source of truth for enabled tools + construction.
- Additions/removals of tools must update SSE mapping, cost accounting, client event types, this DOX when behavior changes.
- `read` and `find` return hashline section: `[#TAG]` snapshot header (TAG = 4-hex content hash from `snapshots.record`) followed by `N:TEXT` rows. `read` takes `offset`/`limit` (1-based paging); `find` takes `text` (+ optional `regex`/`ignoreCase`/`context`/`limit`). Model must copy `#TAG` verbatim into next `edit`. Landing agent runs read/find/edit tools in **tag-only** mode (`{ tagOnly: true }` in `tools/landing-tools.ts`): headers = `[#TAG]` no path, because `HtmlStoreFilesystem` holds single in-memory document, ignores path — the `index.html` segment upstream DSL carries = pure model-facing noise. `formatHashlineHeader(path | undefined, tag)` emits `[path#TAG]` vs `[#TAG]`; `splitPatchInput(diff, { defaultPath })` resolves tag-only header to tool's internal path so snapshot keys stay consistent. Parser still accepts full `[path#TAG]` form, `tagOnly: false` (factory default) restores path-emitting headers for future multi-file use. `edit` takes `{ action?, diff }` where `diff` = hashline DSL — `[#TAG]` header then `SWAP N.=M:` + `+TEXT` body rows (replace lines N..M inclusive), `DEL N.=M` (delete), or `INS.PRE|POST|HEAD|TAIL N:` (insert). Line numbers refer to ORIGINAL file, never shift as hunks apply; touch only lines `read` displayed. Engine (`new Patcher({ fs, snapshots })` in `tools/edit.ts`) verifies tag (`MismatchError` → re-read on stale/fabricated), applies via `prepare` (validate, no write) → `checkHtmlBalance` (rejects unbalanced container tags — e.g. eaten `</style>`/`</head>` — BEFORE write; on failure tool first runs `autofixHtmlBalance`, which (1) collapses immediately-adjacent duplicate open/close tags, (2) for TRUNCATED edit (opens w/o closers, e.g. `maxOutputTokens` cut-off) appends EOF-open tags' closers via clean nesting walk — proceeding w/ `html autofix:` warning when rebalances. Walk aborts on any out-of-order closer, so closer deleted mid-document (eaten `</style>`/`</head>`) still rejects, forces re-read; only genuinely truncated tail repaired) → `commit` (writes via `HtmlStoreFilesystem` → `store.set`, records fresh snapshot). Build incrementally (scaffold shell → tokens → fill sections), each `edit` = complete logical unit (e.g. one section) — never emit full finished page in one edit (`maxOutputTokens` cap mid-generation truncates). At SSE boundary one `edit` call emits exactly one `tool_call` block (1:1 — no `edits.length>=2` fan-out, no `tool_call_drop`), carrying call's top-level `action`; `html_update` fires once per successful edit. One `edit` may batch many `SWAP`/`DEL`/`INS` ops + multiple `[#TAG]` sections anchored to single pre-edit `read`; engine applies in order, mints one fresh `#TAG` for call, so medium-sized logical change = one tool call, not many tiny edits.
- `openrouterModel()` (`lib/openrouter-model.ts`) uses Mastra `OpenAICompatibleConfig` w/ ids shaped as `openrouter/${modelId}` so synthetic prefix stripped, full OpenRouter model id sent verbatim to `openrouter.ai/api/v1/chat/completions`. OpenRouter = only provider for all agent LLM traffic (text brain), image generation, vision OCR. All three roles selectable per request via `/agent { textModel?, imageModel?, visionModel? }`, default from `config.openrouter.default{Chat,Image,Vision}Model`; default chat model + client-provided Text options use OpenRouter's `:nitro` routing variant. Main agent retries configured via `config.agentRetry`; hidden AI SDK model retries default `0` so user-visible Mastra retry events own backoff reporting unless overridden by env.
- `screenshot` tool's `inputSchema` intentionally optional + `.passthrough()`, `execute` runs raw input through `recoverScreenshotArgs` before use. GLM-5.2 intermittently streams `selector`/`viewportSize` wrapped in `<arg_key>`/`<arg_value>` tags that collapse JSON into one mangled key; recovery regex-extracts selector/viewportSize/action (parses JSON-string args) so usable call lands instead of validation error. Keep recovery in `execute` (whole-object `z.preprocess` makes Mastra type tool input as `unknown`).
- Streaming tool-call finalization patched (see root AGENTS + `patches/@mastra__core@1.47.0.patch`): Mastra's actual OpenRouter ESM/CommonJS adapters buffer argument fragments, emit `tool-input-*` + `tool-call` only at end-of-stream, replace initial `{}` placeholder when later cumulative JSON arrives, raise provider-stream error for incomplete final JSON instead of silently coercing to `{}`. Older generic OpenAI-compatible safeguard remains. `lib/openrouter-tool-stream.test.ts` executes both installed adapters vs placeholder/cumulative + truncated SSE streams.
- `resolveModelId()` strips optional `openrouter/` prefix, falls back to **chat** default only. Correct for `textModel`, but `/agent` handler must forward `undefined` for omitted `imageModel`/`visionModel` so `streamLandingAgent`'s `StreamOptions` destructuring defaults apply role-specific image/vision models. Routing image/vision through `resolveModelId` silently substitutes chat model (neither image nor vision capable), 404s at provider when callers send only `textModel`.
- Agent generation sampling pinned in `route.ts` `modelSettings` from `config.agentGeneration` (`config-env.ts`): GLM-5.2 temp defaults `1.0` (`AGENT_TEMPERATURE`), or nucleus sampling via optional `AGENT_TOP_P` — only one ever emitted on wire (Z.ai docs say tune EITHER temp OR top_p, never both). `reasoning_effort` intentionally NOT set: generic OpenAI-compatible path (verified by wire capture) drops `providerOptions.openai.reasoningEffort`, GLM-5.2 defaults to `max` (deep reasoning) when param omitted = recommended coding/agentic setting. `maxOutputTokens` stays `16_384` to keep edits incremental (section-sized, never whole finished page) — large enough for medium-sized cohesive edit, small enough to avoid mid-generation truncation.
- Vision OCR uses configured OpenRouter vision model (default `bytedance-seed/seed-2.0-mini`) w/ Z.AI `ui_to_artifact` system prompt (senior-frontend-engineer/code persona: extract structure, spacing scale, hex colors, typography) sent as `system` message, while per-call user prompts (attachment framing, screenshot action, scrape OCR) = `user` message alongside image parts; image generation uses OpenRouter's `/api/v1/images` endpoint w/ configured image model (default `bytedance-seed/seedream-4.5`). OpenRouter cost accounting must use only provider-reported response metadata (`cost`, `total_cost`, `estimated_cost`, or nested cost objects), must NOT estimate USD from tokens/image counts. LLM streaming must request raw chunks, sum terminal provider cost from every OpenRouter generation/agent step; Mastra `step-finish` usage updates rolling stats snapshot while aggregate terminal usage reconciles final snapshot, because `usage.raw` retains only latest step. When metadata absent, OpenRouter cost = `0`. Firecrawl scrape cost calculated from Firecrawl-reported `creditsUsed` × `config.firecrawl.creditUsd` (default `0.002`, override w/ `FIRECRAWL_CREDIT_USD`). Project reads must NOT normalize legacy zero-cost stats from saved token usage. Every outbound OpenRouter/remote `fetch` bounded by `boundedFetch` (`lib/bounded-fetch.ts`, renamed from `fetchVisionCompletion`): configurable `AbortController` timeout + retry on 5xx/abort (4xx returned unretried). Covers vision OCR (30s×3), image generation (30s×3), scraped-image download (15s×2), so slow/hung provider or external image host fails fast instead of hanging agent stream.
- `screenshot` tool accepts exactly `action` + `selector` (no `viewportSize`). `action` = optional, agent-supplied string, becomes vision `user` message alongside Z.AI `ui_to_artifact` system prompt (e.g. "check hero spacing and CTA contrast"); when omitted defaults to generic inspection prompt. Route's capture callback calls `captureProjectSelectors` from `lib/project-screenshot.ts` over current `store.get()` HTML, connects to Cloudflare Browser Run via remote CDP (`playwright-core`), renders selector at three fixed viewports (mobile 390×844, tablet 768×1024, desktop 1440×900) in one no-JavaScript browser session w/ external requests blocked + same-project images inlined, persists JPEGs (quality 90, 8px white padding, 0.5 scale, 4096 cap). Three viewport images OCR'd in one vision call w/ configured OpenRouter vision model. Compact result reports capture count + OCR image count; route maps all three safe persisted screenshot URLs into terminal `tool_call.images` preview. Numbered red badges + Set-of-Marks element maps generated server-side via trusted CDP `page.evaluate`. `viewportSize` gone from tool schema, SSE events, client contracts. See `lib/project-screenshot.ts` for full capture contract.
- Element attachments carry only `{ kind: 'element', selector }` on wire; uploaded/pasted images still send full payload w/ base64 `dataUrl`. Server's `analyzePromptAttachments` gathers all element selectors, calls one `captureProjectSelectors` batch over current HTML, flattens resulting three viewport images per selector alongside uploaded image data URLs into one bounded OCR call. Safe persisted screenshot URLs for element captures included in terminal `analyze_image` tool `images` preview; uploaded-image data URLs enriched client-side from attachment state (never sent through durable event log). Element attachment metadata persisted to turns = `{ id, kind: 'element', name, selector }` — no `html`, `mediaType`, `screenshotHeight/Width`, or `size`. Legacy persisted turns w/ old element fields hydrate via backward-compatible optional fields in `ConversationAttachment`.
- `design` skill + the planner experiment were both REMOVED: no `design` InlineSkill, no separate planner call, no `LANDING_PAGE_DESIGN_GUIDANCE` (file deleted). Agent has no `skill`/`skill_read`/`skill_search` tools + no design guidance in its context — it implements the user's request directly. `@workspace/agent-skills` remains workspace dep + package (reversible) but no longer imported.
- Image store = process-memory during live `generate_image` call; `createProjectHtmlStore`'s sync write path copies those bytes into `.data/projects/<id>/images/`, rewrites URLs to `/api/projects/:id/images/<file>` as agent edits document. In-memory image store = source only during run; project `html.json` file = source of truth.
