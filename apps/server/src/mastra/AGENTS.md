# Mastra Agent DOX

## Purpose

- Owns the Mastra implementation of the landing-page agent and its custom server-side streaming protocol.

## Ownership

- `agents/`: landing-page agent factory and singleton registration config; the system prompt carries a hashline `read`/`find`/`edit` quick reference and a MANDATORY `plan`-tool-first rule. No `design` skill (removed as an experiment); the landing-page design-system guidance is attached to the session's first user message by `buildAgentMessages` (`route.ts`).
- `tools/`: Mastra tool factories and the landing tool registry, including plan/scrape/read/find/edit/generate_image/screenshot; legacy grep helpers may remain internal but are not public landing tools.
- `lib/`: OpenRouter model config, OpenRouter vision OCR (`image-ocr.ts`) with a bounded fetch helper (`bounded-fetch.ts`), Cloudflare Browser Run screenshot capture (`project-screenshot.ts`) via remote CDP with `playwright-core` (no local browser), cost accounting, anchored HTML document/edit/find/image/SSE helpers, and file-backed project storage (`project-store.ts`) — append-only debug logs (`client-messages.jsonl`, `agent-messages.jsonl`, `vision-messages.json`), screenshot byte persistence, hydration via the shared `@workspace/conversation` reducer (`replayClientEvents`), and `readAgentRawByTurn` (last per-turn Mastra snapshot for replay).
- `route.ts` claims one process-local active run per `projectId` immediately after project lookup and rejects overlap without mutating metadata or appending a second prompt. It records one inbound prompt (`dir:"in"`, `type:"prompt"`) with the supplied client `turnId` (generated fallback only when omitted), then maps Mastra `fullStream` chunks through `emit(event,payload)`: each outbound event is queued to `client-messages.jsonl` before best-effort SSE delivery, so a destroyed/throwing socket cannot prevent replay. Prompt attachment OCR receives the same run abort signal as Mastra generation, including image downloads, provider fetch, and retry delay. Successful content-changing edits emit `html_update`; model retries emit `retry`; tool errors/malformed edit results emit terminal error tool states. Normal edit failures remain recoverable until `MAX_EDIT_FAILURES`; the optional cost cap remains fatal. Cost/stats accounting emits rolling snapshots after provider cost metadata, every completed LLM step, attachment analysis, and each terminal tool event; a final snapshot still runs in `finally` for success, stream throws, and aborts, using only provider-reported values. Non-fatal aborts retain `finishReason:"stopped"` even if Mastra reports `tripwire`. Accepted runs persist terminal `stats`, optional `error`/stopped, then `done`; unconditional outer cleanup conditionally releases the registry owner, flushes logs, and best-effort ends the response regardless of socket writes. `stopLandingAgent(projectId)` aborts attachment/Mastra/screenshot-capture work without closing SSE. Client SSE/browser disconnect is delivery loss, not cancellation — accepted runs continue in the same server process and persist terminal events regardless. Agent messages are snapshotted per step and at run end, generated images are persisted at tool-result time, and an unchanged draft still gets the empty-draft terminal error.
- Attachment vision OCR passes the active run signal into `boundedFetch`; it composes external cancellation with each per-attempt timeout, cancels retry sleep, and propagates external abort immediately while preserving existing 5xx/internal-timeout retries.
- `index.ts`: Mastra instance, storage, logger, and observability setup.

## Local Contracts

- The agent edits the project HTML via a hashline edit engine (first-party, vendored under `lib/hashline/`): `read`/`find` emit a `[index.html#TAG]` snapshot header + `N:TEXT` rows and record a content-hash snapshot; `edit` takes `{ action, diff }` (hashline DSL), verifies the `#TAG` against the snapshot (rejects stale/fabricated tags with a re-read instruction), runs an HTML-tag-balance guard, then writes. The store is the write-through `createProjectHtmlStore` over `html.json` (the persisted source of truth); tools never mutate repo files and the UI never writes HTML. `index.html` is legacy import-only. Initial/fallback reads use `getProject`; live edits stream rendered `html_update` SSE payloads (not persisted into turns). The `lib/hashline/` core is exempt from the project's perfectionist sort + `no-explicit-any` style rules (`apps/server/oxlint.config.ts` override); its unused vendored surface was deleted down to what this consumer uses, so `fallow:dead-code` passes with no exclude. First-party, no external attribution.
- The project title is set server-side from the first prompt (`setTitleIfUntitled`) on `POST /agent`; `ProjectMeta` persists all three per-role models (`textModel`/`imageModel`/`visionModel`) via `PATCH /api/projects/:id`, while `POST /agent` run-start persists only `textModel` and each saved message turn keeps the text model used for that turn. All `project.json` read-modify-writes are synchronous (`readMetaSync`/`writeMetaSync`) so a model PATCH serializes against edit-time `markHasHtmlSync` writes with no lost update (no `hasHtml` reversion).
- Project conversation history is server-owned and persisted as append-only logs, each holding one execution context and appended per event/step (never overwritten) so the data at any moment is inspectable: (1) `client-messages.jsonl` — the client wire (`dir:"out"` = each SSE event; `dir:"in"` = prompt + screenshot POST-back, metadata + file pointer only, never base64); `getProject` `replayClientEvents`-reduces it into `ProjectMessageTurn[]` for hydration (each `dir:"in"` prompt starts a turn with the real `turnId`; `done` terminalizes; tools left running are terminalized to `error` on restore). (2) `agent-messages.jsonl` — per-step Mastra `messageList` snapshots (`stream.messageList.get.response.db()` via `onStepFinish` + once at run end), with `reasoning` parts stripped (`stripReasoning`) so replay doesn't inflate the next prompt; `readAgentRawByTurn` returns the last snapshot per turn. Both fall back to legacy `messages.json`/`raw-messages.json` for old projects. `streamLandingAgent` replays the agent snapshots verbatim into the next OpenRouter agent call so the model sees its previous tool calls/results instead of a lossy prose paraphrase; it falls back to prose only when a turn has no agent messages. (3) `vision-messages.json` — per OCR/vision call (attachment `ocrImageInputs` AND scrape `ocrImages`, both carrying `projectId`/`turnId` threaded through `LandingToolContext`), text/usage/cost only. (4) `screenshots/<NNN>-<requestId>.<ext>` — captured screenshot bytes written at POST-back resolve. Prompt image attachments are OCR analyzed before the OpenRouter agent run; selected-element attachments also pass their captured `outerHTML` into agent context alongside screenshot OCR. No base64 image bytes are ever written to a JSON log.
- User-facing agent text should stay concise, use the latest user prompt's language (English when English or ambiguous), and must not echo internal tool transcripts such as `Tool read done`, `Action:`, `Detail:`, or `Result:`; the UI renders tool status separately.
- `LANDING_AGENT_INSTRUCTIONS` (`agents/landing-page-agent.ts`) includes the advisory `HASHLINE_SYSTEM_GUIDANCE` quick reference from `lib/hashline/edit-prompt.ts`: snapshot headers/rows, `SWAP`/`DEL`/`INS`, `+` insertion-row encoding (including CSS `--variables`), balanced closing tags, fresh tags, and failure recovery. It also keeps fix/refine requests to the smallest satisfying change to avoid over-production; the `maxOutputTokens` cap truncates over-long generations with `finishReason=length`. It also enforces a MANDATORY `plan`-tool-first rule for new/redesign/substantial work; the full bastrb-derived landing-page design-system guidance is NOT in the system prompt — it is prepended to the session's first `user` message by `buildAgentMessages` from `LANDING_PAGE_DESIGN_GUIDANCE` (`lib/landing-design-guidance.ts`), reconstructed on every turn's replay so it persists without re-injection and never reaches the UI-facing `turn.prompt`.
- Every user-visible tool call must include an `action`; the client renders it in the conversation UI. Tools that accept an `action` should receive one from the agent. The `design` skill (and its `skill`/`skill_read`/`skill_search` tools) is removed; the route's `skill*` display branches remain as harmless dead code. `plan` has no `action` arg — `defaultToolAction` labels it `Plan` and the UI renders it via the dedicated `PlanBlock` (`packages/prompt-panel/src/turn-steps.tsx`) from its result actions, not as a normal tool row.
- Mastra `tool-result` chunks with top-level `{ ok: false }` are terminal failures for the custom SSE protocol and must stream and record `tool_call.state = "error"` even when the SDK chunk `isError` flag is false.
- `tools/landing-tools.ts` is the source of truth for enabled tools and their construction.
- The `plan` tool (`tools/plan.ts`) is a pure-reasoning, MANDATORY-first tool: the system prompt forces a `plan({ actions, request })` call before any new page, redesign, or substantial change. It makes no provider/external call (no `boundedFetch`, no cost accounting), returns `{ actions, message: "implement this now", ok }`, and is wired with no context (no fs/snapshots/store). At the SSE boundary the route joins the echoed result `actions` into the `tool_call` `result` (`summarizeResultForTool`) and labels the call `Plan` (`defaultToolAction`); the UI `PlanBlock` renders those actions as a numbered list, so `plan` needs no new SSE event and no client payload change.
- Additions or removals of tools must update SSE mapping, cost accounting, client event types, and this DOX when behavior changes.
- `read` and `find` return a hashline section: a `[index.html#TAG]` snapshot header (TAG = 4-hex content hash from `snapshots.record`) followed by `N:TEXT` rows. `read` takes `offset`/`limit` (1-based paging); `find` takes `text` (+ optional `regex`/`ignoreCase`/`context`/`limit`). The model must copy the `#TAG` verbatim into its next `edit`. `edit` takes `{ action?, diff }` where `diff` is the hashline DSL — the `[index.html#TAG]` header then `SWAP N.=M:` + `+TEXT` body rows (replace lines N..M inclusive), `DEL N.=M` (delete), or `INS.PRE|POST|HEAD|TAIL N:` (insert). Line numbers refer to the ORIGINAL file and never shift as hunks apply; touch only lines `read` displayed. The engine (`new Patcher({ fs, snapshots })` in `tools/edit.ts`) verifies the tag (`MismatchError` → re-read on stale/fabricated), applies via `prepare` (validate, no write) → `checkHtmlBalance` (rejects unbalanced container tags — e.g. an eaten `</style>`/`</head>` — BEFORE write) → `commit` (writes via `HtmlStoreFilesystem` → `store.set`, records the fresh snapshot). Build incrementally (scaffold shell → tokens → fill sections); never emit a full finished page in one edit (`maxOutputTokens` cap mid-generation truncates). At the SSE boundary one `edit` call emits exactly one `tool_call` block (1:1 — no `edits.length>=2` fan-out, no `tool_call_drop`), carrying the call's top-level `action`; `html_update` fires once per successful edit.
- `openrouterModel()` (`lib/openrouter-model.ts`) uses Mastra `OpenAICompatibleConfig` with ids shaped as `openrouter/${modelId}` so the synthetic prefix is stripped and the full OpenRouter model id is sent verbatim to `openrouter.ai/api/v1/chat/completions`. OpenRouter is the only provider for all agent LLM traffic (text brain), image generation, and vision OCR. All three roles are selectable per request via `/agent { textModel?, imageModel?, visionModel? }` and default from `config.openrouter.default{Chat,Image,Vision}Model`; the default chat model and client-provided Text options use OpenRouter's `:nitro` routing variant. Main agent retries are configured through `config.agentRetry`; hidden AI SDK model retries default to `0` so user-visible Mastra retry events own backoff reporting unless overridden by env.
- The `screenshot` tool's `inputSchema` is intentionally optional + `.passthrough()` and `execute` runs the raw input through `recoverScreenshotArgs` before use. GLM-5.2 intermittently streams `selector`/`viewportSize` wrapped in `<arg_key>`/`<arg_value>` tags that collapse the JSON into one mangled key; recovery regex-extracts selector/viewportSize/action (and parses JSON-string args) so a usable call lands instead of a validation error. Keep the recovery in `execute` (a whole-object `z.preprocess` makes Mastra type the tool input as `unknown`).
- Streaming tool-call finalization is patched (see root AGENTS + `patches/@mastra__core@1.47.0.patch`): Mastra's actual OpenRouter ESM/CommonJS adapters buffer argument fragments and emit `tool-input-*` plus `tool-call` only at end-of-stream, replace an initial `{}` placeholder when later cumulative JSON arrives, and raise a provider-stream error for incomplete final JSON instead of silently coercing it to `{}`. The older generic OpenAI-compatible safeguard remains. `lib/openrouter-tool-stream.test.ts` executes both installed adapters against placeholder/cumulative and truncated SSE streams.
- `resolveModelId()` strips an optional `openrouter/` prefix and falls back to the **chat** default only. It is correct for `textModel`, but the `/agent` handler must forward `undefined` for omitted `imageModel`/`visionModel` so `streamLandingAgent`'s `StreamOptions` destructuring defaults apply the role-specific image/vision models. Routing image/vision through `resolveModelId` silently substitutes the chat model (neither image nor vision capable) and 404s at the provider when callers send only `textModel`.
- Agent generation sampling is pinned in `route.ts` `modelSettings` from `config.agentGeneration` (`config-env.ts`): GLM-5.2 temperature defaults to `1.0` (`AGENT_TEMPERATURE`), or nucleus sampling via optional `AGENT_TOP_P` — only one is ever emitted on the wire (Z.ai docs say tune EITHER temperature OR top_p, never both). `reasoning_effort` is intentionally NOT set: the generic OpenAI-compatible path (verified by wire capture) drops `providerOptions.openai.reasoningEffort`, and GLM-5.2 defaults to `max` (deep reasoning) when the param is omitted, which is the recommended coding/agentic setting. `maxOutputTokens` stays `16_384` to force incremental hashline edits.
- Vision OCR uses the configured OpenRouter vision model (default `bytedance-seed/seed-2.0-mini`) with the Z.AI `ui_to_artifact` system prompt (the senior-frontend-engineer/code persona: extract structure, spacing scale, hex colors, and typography) sent as the `system` message, while per-call user prompts (attachment framing, screenshot action, scrape OCR) are the `user` message alongside the image parts; image generation uses OpenRouter's `/api/v1/images` endpoint with the configured image model (default `bytedance-seed/seedream-4.5`). OpenRouter cost accounting must use only provider-reported response metadata (`cost`, `total_cost`, `estimated_cost`, or nested cost objects) and must not estimate USD from tokens or image counts. LLM streaming must request raw chunks and sum the terminal provider cost from every OpenRouter generation/agent step; Mastra `step-finish` usage updates the rolling stats snapshot while aggregate terminal usage reconciles the final snapshot, because `usage.raw` retains only the latest step. When metadata is absent, the OpenRouter cost is `0`. Firecrawl scrape cost is calculated from Firecrawl-reported `creditsUsed` multiplied by `config.firecrawl.creditUsd` (default `0.002`, override with `FIRECRAWL_CREDIT_USD`). Project reads must not normalize legacy zero-cost stats from saved token usage. Every outbound OpenRouter/remote `fetch` is bounded by `boundedFetch` (`lib/bounded-fetch.ts`, renamed from `fetchVisionCompletion`): a configurable `AbortController` timeout + retry on 5xx/abort (4xx returned unretried). It covers vision OCR (30s×3), image generation (30s×3), and scraped-image download (15s×2), so a slow or hung provider or external image host fails fast instead of hanging the agent stream.
- The `screenshot` tool accepts exactly `action` and `selector` (no `viewportSize`). `action` is an optional, agent-supplied string that becomes the vision `user` message alongside the Z.AI `ui_to_artifact` system prompt (e.g. "check hero spacing and CTA contrast"); when omitted it defaults to a generic inspection prompt. The route's capture callback calls `captureProjectSelectors` from `lib/project-screenshot.ts` over current `store.get()` HTML, which connects to Cloudflare Browser Run via remote CDP (`playwright-core`), renders the selector at three fixed viewports (mobile 390×844, tablet 768×1024, desktop 1440×900) in one no-JavaScript browser session with external requests blocked and same-project images inlined, and persists JPEGs (quality 90, 8px white padding, 0.5 scale, 4096 cap). The three viewport images are OCR'd in one vision call with the configured OpenRouter vision model. The compact result reports the capture count and OCR image count; the route maps all three safe persisted screenshot URLs into the terminal `tool_call.images` preview. Numbered red badges and Set-of-Marks element maps are generated server-side via trusted CDP `page.evaluate`. `viewportSize` is gone from the tool schema, SSE events, and client contracts. See `lib/project-screenshot.ts` for the full capture contract.
- Element attachments carry only `{ kind: 'element', selector }` on the wire; uploaded/pasted images still send their full payload with a base64 `dataUrl`. The server's `analyzePromptAttachments` gathers all element selectors and calls one `captureProjectSelectors` batch over current HTML, then flattens the resulting three viewport images per selector alongside uploaded image data URLs into one bounded OCR call. Safe persisted screenshot URLs for element captures are included in the terminal `analyze_image` tool `images` preview; uploaded-image data URLs are enriched client-side from the attachment state (never sent through the durable event log). Element attachment metadata persisted to turns is `{ id, kind: 'element', name, selector }` — no `html`, `mediaType`, `screenshotHeight/Width`, or `size`. Legacy persisted turns with the old element fields hydrate via backward-compatible optional fields in `ConversationAttachment`.
- The `design` skill was REMOVED (experiment): the landing-page design-system guidance now travels on the session's FIRST user message (`LANDING_PAGE_DESIGN_GUIDANCE`, prepended by `buildAgentMessages`) instead of a Mastra InlineSkill, so it is paid once and stays out of the per-request system prompt. The agent has no `skill`/`skill_read`/`skill_search` tools. `@workspace/agent-skills` remains a workspace dep + package (reversible) but is no longer imported; re-enabling means restoring `import { design } from '@workspace/agent-skills'`, `skills: [design]`, and dropping the first-message guidance prepend.
- The image store is process-memory during a live `generate_image` call; `createProjectHtmlStore`'s sync write path copies those bytes into `.data/projects/<id>/images/` and rewrites their URLs to `/api/projects/:id/images/<file>` as the agent edits the document. The in-memory image store is source only during a run; the project `html.json` file is the source of truth.

## Work Guidance

- For Mastra changes, read installed package docs/types first; Mastra APIs change quickly.
- Keep tests off system-prompt prose. Test the hashline parser/tools as executable behavior and use live traces to judge model adherence.
- Keep per-request `Agent` + `HtmlStore` isolation for production requests; the singleton registration exists for Mastra Studio visibility.
- Prefer `zod` schemas at tool boundaries and small pure helpers in `lib/`.
