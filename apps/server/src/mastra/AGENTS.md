# Mastra Agent DOX

## Purpose

- Owns the Mastra implementation of the landing-page agent and its custom server-side streaming protocol.

## Ownership

- `agents/`: landing-page agent factory and singleton registration config; attaches the verbatim pi `design` skill from `@workspace/agent-skills` via `skills: [design]`.
- `tools/`: Mastra tool factories and the landing tool registry, including scrape/read/find/edit/generate_image/screenshot; legacy grep helpers may remain internal but are not public landing tools.
- `lib/`: OpenRouter model config, OpenRouter vision OCR (`image-ocr.ts`) with a bounded fetch helper (`bounded-fetch.ts`), browser screenshot response registry, cost accounting, anchored HTML document/edit/find/image/SSE helpers, and file-backed project storage (`project-store.ts`) â€” append-only debug logs (`client-messages.jsonl`, `agent-messages.jsonl`, `vision-messages.json`), screenshot byte persistence, hydration via the shared `@workspace/conversation` reducer (`replayClientEvents`), and `readAgentRawByTurn` (last per-turn Mastra snapshot for replay).
- `route.ts` maps Mastra `fullStream` chunks to the client-facing SSE protocol via an `emit(event,payload)` wrapper that BOTH streams the SSE frame AND appends it (timestamped, `dir:"out"`) to `client-messages.jsonl`, so the exact client-visible data at any moment is inspectable mid-run. The inbound prompt is recorded once at run start (`dir:"in"`, `type:"prompt"`, carrying the real `turnId`). It accepts `projectId`, validates the project exists, sets the title from the prompt, and builds a project-scoped write-through store. After successful content-changing `edit` results, it streams `html_update` with project id, sequence, previous/current hashes, byte count, and current server-rendered HTML so the client can morph the preview without a full project fetch. Mastra-level model retries emit a `retry` SSE event with the issue, attempt, max attempts, and backoff delay before waiting and retrying. Mastra `tool-error` chunks must be surfaced as terminal `tool_call` events with `state: "error"` and the error message. Null or malformed `edit` results are tool errors, not successful edits. A failed `edit` does NOT abort the run â€” the model receives the real edit error as feedback and may recover (e.g. by `read`/`find`); only `MAX_EDIT_FAILURES` (10) repeated edit failures stop the run. An optional per-run cost cap (`AGENT_MAX_COST_USD`, checked after each LLM/image/vision cost accrual) also hard-aborts the run with an `error` SSE event when met. Cost/stats accounting runs unconditionally after the stream loop (even on a fatal/aborted run), guarded against `stream.usage`/`finishReason` throwing on an aborted stream. Agent messages are snapshotted per step via `onStepFinish` (`stream.messageList.get.response.db()`, reasoning stripped) AND once at run end into `agent-messages.jsonl`. Generated images are persisted to the project `images/` folder at `generate_image` tool-result time (via `persistGeneratedImage`), independent of a later successful edit, so bytes are never orphaned. `flushProjectLogs` (awaited in `finally` before `response.end()`) drains the per-project write chain so logs are durable. If a draft project finishes without a successful HTML-generating edit, the stream emits a terminal `error` event after stats instead of reporting a successful empty page.
- `index.ts`: Mastra instance, storage, logger, and observability setup.

## Local Contracts

- The agent edits the project HTML via a hashline edit engine (first-party, vendored under `lib/hashline/`): `read`/`find` emit a `[index.html#TAG]` snapshot header + `N:TEXT` rows and record a content-hash snapshot; `edit` takes `{ action, diff }` (hashline DSL), verifies the `#TAG` against the snapshot (rejects stale/fabricated tags with a re-read instruction), runs an HTML-tag-balance guard, then writes. The store is the write-through `createProjectHtmlStore` over `html.json` (the persisted source of truth); tools never mutate repo files and the UI never writes HTML. `index.html` is legacy import-only. Initial/fallback reads use `getProject`; live edits stream rendered `html_update` SSE payloads (not persisted into turns). The `lib/hashline/` core is exempt from the project's perfectionist sort + `no-explicit-any` style rules (`apps/server/oxlint.config.ts` override); its unused vendored surface was deleted down to what this consumer uses, so `fallow:dead-code` passes with no exclude. First-party, no external attribution.
- The project title is set server-side from the first prompt (`setTitleIfUntitled`) on `POST /agent`; `ProjectMeta` persists all three per-role models (`textModel`/`imageModel`/`visionModel`) via `PATCH /api/projects/:id`, while `POST /agent` run-start persists only `textModel` and each saved message turn keeps the text model used for that turn. All `project.json` read-modify-writes are synchronous (`readMetaSync`/`writeMetaSync`) so a model PATCH serializes against edit-time `markHasHtmlSync` writes with no lost update (no `hasHtml` reversion).
- Project conversation history is server-owned and persisted as append-only logs, each holding one execution context and appended per event/step (never overwritten) so the data at any moment is inspectable: (1) `client-messages.jsonl` â€” the client wire (`dir:"out"` = each SSE event; `dir:"in"` = prompt + screenshot POST-back, metadata + file pointer only, never base64); `getProject` `replayClientEvents`-reduces it into `ProjectMessageTurn[]` for hydration (each `dir:"in"` prompt starts a turn with the real `turnId`; `done` terminalizes; tools left running are terminalized to `error` on restore). (2) `agent-messages.jsonl` â€” per-step Mastra `messageList` snapshots (`stream.messageList.get.response.db()` via `onStepFinish` + once at run end), with `reasoning` parts stripped (`stripReasoning`) so replay doesn't inflate the next prompt; `readAgentRawByTurn` returns the last snapshot per turn. Both fall back to legacy `messages.json`/`raw-messages.json` for old projects. `streamLandingAgent` replays the agent snapshots verbatim into the next OpenRouter agent call so the model sees its previous tool calls/results instead of a lossy prose paraphrase; it falls back to prose only when a turn has no agent messages. (3) `vision-messages.json` â€” per OCR/vision call (attachment `ocrImageInputs` AND scrape `ocrImages`, both carrying `projectId`/`turnId` threaded through `LandingToolContext`), text/usage/cost only. (4) `screenshots/<NNN>-<requestId>.<ext>` â€” captured screenshot bytes written at POST-back resolve. Prompt image attachments are OCR analyzed before the OpenRouter agent run; selected-element attachments also pass their captured `outerHTML` into agent context alongside screenshot OCR. No base64 image bytes are ever written to a JSON log.
- User-facing agent text should stay concise, use the latest user prompt's language (English when English or ambiguous), and must not echo internal tool transcripts such as `Tool read done`, `Action:`, `Detail:`, or `Result:`; the UI renders tool status separately.
- `LANDING_AGENT_INSTRUCTIONS` (`agents/landing-page-agent.ts`) directs the agent to make the smallest change that satisfies a fix/refine request â€” no redesigning or regenerating sections the user did not mention â€” to avoid over-production; the `maxOutputTokens` cap truncates over-long generations with `finishReason=length`.
- Every user-visible tool call must include an `action`; the client renders it in the conversation UI. Tools that accept an `action` should receive one from the agent. `skill` and `skill_read` derive intents from `name`/`skillName`+`path` because their schemas have no `action` field.
- Mastra `tool-result` chunks with top-level `{ ok: false }` are terminal failures for the custom SSE protocol and must stream and record `tool_call.state = "error"` even when the SDK chunk `isError` flag is false.
- `tools/landing-tools.ts` is the source of truth for enabled tools, tool count/list, and tool guidance.
- Additions or removals of tools must update SSE mapping, cost accounting, client event types, and this DOX when behavior changes.
- `read` and `find` return a hashline section: a `[index.html#TAG]` snapshot header (TAG = 4-hex content hash from `snapshots.record`) followed by `N:TEXT` rows. `read` takes `offset`/`limit` (1-based paging); `find` takes `text` (+ optional `regex`/`ignoreCase`/`context`/`limit`). The model must copy the `#TAG` verbatim into its next `edit`. `edit` takes `{ action?, diff }` where `diff` is the hashline DSL â€” the `[index.html#TAG]` header then `SWAP N.=M:` + `+TEXT` body rows (replace lines N..M inclusive), `DEL N.=M` (delete), or `INS.PRE|POST|HEAD|TAIL N:` (insert). Line numbers refer to the ORIGINAL file and never shift as hunks apply; touch only lines `read` displayed. The engine (`new Patcher({ fs, snapshots })` in `tools/edit.ts`) verifies the tag (`MismatchError` â†’ re-read on stale/fabricated), applies via `prepare` (validate, no write) â†’ `checkHtmlBalance` (rejects unbalanced container tags â€” e.g. an eaten `</style>`/`</head>` â€” BEFORE write) â†’ `commit` (writes via `HtmlStoreFilesystem` â†’ `store.set`, records the fresh snapshot). Build incrementally (scaffold shell â†’ tokens â†’ fill sections); never emit a full finished page in one edit (`maxOutputTokens` cap mid-generation truncates). At the SSE boundary one `edit` call emits exactly one `tool_call` block (1:1 â€” no `edits.length>=2` fan-out, no `tool_call_drop`), carrying the call's top-level `action`; `html_update` fires once per successful edit.
- `openrouterModel()` (`lib/openrouter-model.ts`) uses Mastra `OpenAICompatibleConfig` with ids shaped as `openrouter/${modelId}` so the synthetic prefix is stripped and the full OpenRouter model id is sent verbatim to `openrouter.ai/api/v1/chat/completions`. OpenRouter is the only provider for all agent LLM traffic (text brain), image generation, and vision OCR. All three roles are selectable per request via `/agent { textModel?, imageModel?, visionModel? }` and default from `config.openrouter.default{Chat,Image,Vision}Model`. Main agent retries are configured through `config.agentRetry`; hidden AI SDK model retries default to `0` so user-visible Mastra retry events own backoff reporting unless overridden by env.
- The `screenshot` tool's `inputSchema` is intentionally optional + `.passthrough()` and `execute` runs the raw input through `recoverScreenshotArgs` before use. GLM-5.2 intermittently streams `selector`/`viewportSize` wrapped in `<arg_key>`/`<arg_value>` tags that collapse the JSON into one mangled key; recovery regex-extracts selector/viewportSize/action (and parses JSON-string args) so a usable call lands instead of a validation error. Keep the recovery in `execute` (a whole-object `z.preprocess` makes Mastra type the tool input as `unknown`).
- Streaming tool-call finalization is patched (see root AGENTS + `patches/@mastra__core@1.47.0.patch`): the `tool_call` SSE event for a step now emits at end-of-stream (all argument fragments accumulated), not mid-stream on the AI-SDK's `isParsableJson` early-finalize. This is what stops GLM-5.2 from dropping multi-fragment `edit` arguments as empty `{}`. Incremental `tool-input-delta` UI streaming is unchanged.
- `resolveModelId()` strips an optional `openrouter/` prefix and falls back to the **chat** default only. It is correct for `textModel`, but the `/agent` handler must forward `undefined` for omitted `imageModel`/`visionModel` so `streamLandingAgent`'s `StreamOptions` destructuring defaults apply the role-specific image/vision models. Routing image/vision through `resolveModelId` silently substitutes the chat model (neither image nor vision capable) and 404s at the provider when callers send only `textModel`.
- Vision OCR uses the configured OpenRouter vision model (default `z-ai/glm-5v-turbo`) with the Z.AI `ui_to_artifact` system prompt (the senior-frontend-engineer/code persona: extract structure, spacing scale, hex colors, and typography) sent as the `system` message, while per-call user prompts (attachment framing, screenshot action, scrape OCR) are the `user` message alongside the image parts; image generation uses OpenRouter's `/api/v1/images` endpoint with the configured image model (default `bytedance-seed/seedream-4.5`). OpenRouter cost accounting must use only provider-reported response metadata (`cost`, `total_cost`, `estimated_cost`, or nested cost objects) and must not estimate USD from tokens or image counts. LLM streaming must request raw chunks and capture provider cost metadata when present; when metadata is absent, the OpenRouter cost is `0`. Firecrawl scrape cost is calculated from Firecrawl-reported `creditsUsed` multiplied by `config.firecrawl.creditUsd` (default `0.002`, override with `FIRECRAWL_CREDIT_USD`). Project reads must not normalize legacy zero-cost stats from saved token usage. Every outbound OpenRouter/remote `fetch` is bounded by `boundedFetch` (`lib/bounded-fetch.ts`, renamed from `fetchVisionCompletion`): a configurable `AbortController` timeout + retry on 5xx/abort (4xx returned unretried). It covers vision OCR (30sĂ—3), image generation (30sĂ—3), and scraped-image download (15sĂ—2), so a slow or hung provider or external image host fails fast instead of hanging the agent stream.
- The `screenshot` tool accepts exactly `action`, `selector`, and `viewportSize` (`mobile` | `tablet` | `desktop`). `action` is an optional, agent-supplied string that becomes the vision `user` message alongside the Z.AI `ui_to_artifact` system prompt (e.g. "check hero spacing and CTA contrast"); when omitted it defaults to a generic inspection prompt. The tool emits a project-correlated `screenshot_request` SSE event, waits on the process-local browser screenshot registry, OCRs the returned selector screenshot with the configured OpenRouter vision model, and returns visual QA notes without creating files or auxiliary artifacts. Because `action` is now a schema field, its displayed reason comes from the agent's `action`; the previous selector/viewport-derived action no longer applies. Client capture renders the requested viewport and adds 8px padding around the selected element image. The client capture also annotates the screenshot with numbered red badges (snapdom `agent-map` plugin, `image:'annotated'`, via the instance `snapdom(el,{plugins})` API) and returns an `elementMap` string (Set-of-Marks: one line per interactive element, `index â†’ role / accessible-name / bbox / state`) alongside the OCR transcript, so the agent can reference interactive elements by badge index. The request callback memoizes timeouts: once a `screenshot_request` times out (no browser client captured it), subsequent `screenshot` calls in the same run fail fast with an actionable reason ("No browser client captured the previous screenshot request...") instead of waiting through the full timeout on every retry.
- The design skill is the landing-page `design` skill from `@workspace/agent-skills` (`design`, a Mastra `InlineSkill`), forked from pi and scoped to landing pages with no file/report/CLI machinery (diagnoses run in memory, fixes apply via `edit`). It complements `LANDING_AGENT_INSTRUCTIONS` and the `read`/`find`/`edit` tool guidance, which own the hashline DSL, incremental build, and "no markdown mockups" rules.
- The image store is process-memory during a live `generate_image` call; `createProjectHtmlStore`'s sync write path copies those bytes into `.data/projects/<id>/images/` and rewrites their URLs to `/api/projects/:id/images/<file>` as the agent edits the document. The in-memory image store is source only during a run; the project `html.json` file is the source of truth.

- Prefer `zod` schemas at tool boundaries and small pure helpers in `lib/`.
