# Mastra Agent DOX

## Purpose

- Owns the Mastra implementation of the landing-page agent and its custom server-side streaming protocol.

## Ownership

- `agents/`: landing-page agent factory and singleton registration config.
- `tools/`: Mastra tool factories and the landing tool registry, including scrape/read/find/edit/generate_image/screenshot; legacy grep helpers may remain internal but are not public landing tools.
- `skills/`: self-contained browser design guidance for the landing-page agent.
- `lib/`: OpenRouter model config, OpenRouter vision OCR, browser screenshot response registry, cost accounting, anchored HTML document/edit/find/image/SSE helpers, and file-backed project storage (`project-store.ts`) — append-only debug logs (`client-messages.jsonl`, `agent-messages.jsonl`, `vision-messages.json`), screenshot byte persistence, hydration via the shared `@workspace/conversation` reducer (`replayClientEvents`), and `readAgentRawByTurn` (last per-turn Mastra snapshot for replay).
- `route.ts` maps Mastra `fullStream` chunks to the client-facing SSE protocol via an `emit(event,payload)` wrapper that BOTH streams the SSE frame AND appends it (timestamped, `dir:"out"`) to `client-messages.jsonl`, so the exact client-visible data at any moment is inspectable mid-run. The inbound prompt is recorded once at run start (`dir:"in"`, `type:"prompt"`, carrying the real `turnId`). It accepts `projectId`, validates the project exists, sets the title from the prompt, and builds a project-scoped write-through store. After successful content-changing `edit` results, it streams `html_update` with project id, sequence, previous/current hashes, byte count, and current server-rendered HTML so the client can morph the preview without a full project fetch. Mastra-level model retries emit a `retry` SSE event with the issue, attempt, max attempts, and backoff delay before waiting and retrying. Mastra `tool-error` chunks must be surfaced as terminal `tool_call` events with `state: "error"` and the error message. Null or malformed `edit` results are tool errors, not successful edits. A failed `edit` does NOT abort the run — the model receives the real edit error as feedback and may recover (e.g. by `read`/`find`); only `MAX_EDIT_FAILURES` (10) repeated edit failures stop the run. An optional per-run cost cap (`AGENT_MAX_COST_USD`, checked after each LLM/image/vision cost accrual) also hard-aborts the run with an `error` SSE event when met. Cost/stats accounting runs unconditionally after the stream loop (even on a fatal/aborted run), guarded against `stream.usage`/`finishReason` throwing on an aborted stream. Agent messages are snapshotted per step via `onStepFinish` (`stream.messageList.get.response.db()`, reasoning stripped) AND once at run end into `agent-messages.jsonl`. Generated images are persisted to the project `images/` folder at `generate_image` tool-result time (via `persistGeneratedImage`), independent of a later successful edit, so bytes are never orphaned. `flushProjectLogs` (awaited in `finally` before `response.end()`) drains the per-project write chain so logs are durable. If a draft project finishes without a successful HTML-generating edit, the stream emits a terminal `error` event after stats instead of reporting a successful empty page.
- `index.ts`: Mastra instance, storage, logger, and observability setup.

## Local Contracts

- The agent edits the project's anchored `html.json` document via a write-through store (`createProjectHtmlStore`). Tools must not mutate repository files, and `html.json` is the single source of truth — the UI never writes HTML. `index.html` is legacy import-only input and must not be written as a mirror. Initial/fallback project reads use `getProject`; live edit updates use rendered `html_update` SSE payloads derived from the write-through store and are not persisted into message turns.
- The project title is set server-side from the first prompt (`setTitleIfUntitled`) on `POST /agent`; the selected model is persisted to project metadata at run start while each saved message turn keeps the model used for that turn.
- Project conversation history is server-owned and persisted as append-only logs, each holding one execution context and appended per event/step (never overwritten) so the data at any moment is inspectable: (1) `client-messages.jsonl` — the client wire (`dir:"out"` = each SSE event; `dir:"in"` = prompt + screenshot POST-back, metadata + file pointer only, never base64); `getProject` `replayClientEvents`-reduces it into `ProjectMessageTurn[]` for hydration (each `dir:"in"` prompt starts a turn with the real `turnId`; `done` terminalizes; tools left running are terminalized to `error` on restore). (2) `agent-messages.jsonl` — per-step Mastra `messageList` snapshots (`stream.messageList.get.response.db()` via `onStepFinish` + once at run end), with `reasoning` parts stripped (`stripReasoning`) so replay doesn't inflate the next prompt; `readAgentRawByTurn` returns the last snapshot per turn. Both fall back to legacy `messages.json`/`raw-messages.json` for old projects. `streamLandingAgent` replays the agent snapshots verbatim into the next OpenRouter agent call so the model sees its previous tool calls/results instead of a lossy prose paraphrase; it falls back to prose only when a turn has no agent messages. (3) `vision-messages.json` — per OCR/vision call (attachment `ocrImageInputs` AND scrape `ocrImages`, both carrying `projectId`/`turnId` threaded through `LandingToolContext`), text/usage/cost only. (4) `screenshots/<NNN>-<requestId>.<ext>` — captured screenshot bytes written at POST-back resolve. Prompt image attachments are OCR analyzed before the OpenRouter agent run; selected-element attachments also pass their captured `outerHTML` into agent context alongside screenshot OCR. No base64 image bytes are ever written to a JSON log.
- User-facing agent text should stay concise, use the latest user prompt's language (English when English or ambiguous), and must not echo internal tool transcripts such as `Tool read done`, `Action:`, `Detail:`, or `Result:`; the UI renders tool status separately.
- Every user-visible tool call must include an `action`; the client renders it in the conversation UI. Tools that accept an `action` should receive one from the agent. `skill` and `skill_read` derive intents from `name`/`skillName`+`path` because their schemas have no `action` field.
- Mastra `tool-result` chunks with top-level `{ ok: false }` are terminal failures for the custom SSE protocol and must stream and record `tool_call.state = "error"` even when the SDK chunk `isError` flag is false.
- `tools/landing-tools.ts` is the source of truth for enabled tools, tool count/list, and tool guidance.
- Additions or removals of tools must update SSE mapping, cost accounting, client event types, and this DOX when behavior changes.
- `read` and `find` return compact anchored lines as `anchor|text` plus checksum/line metadata. `read` takes `from`/`to` (a real anchor or the `start`/`end` document-boundary sentinels, order-insensitive) and `limit`; the positional `range` and line-number `offset` args are removed. `find` defaults to literal search and supports optional regex/context. `edit` must accept `edits: [{ action?, from?, to?, code?, insert? }]` batches — each edit carries its own optional `action` (a one-line label shown to the user; there is no call-level action, and omitting it just yields a blank label rather than erroring). Fields discriminate by presence: omit `from`/`to` and give `code` to replace the whole document (scaffold shell only — the create workflow builds incrementally: scaffold empty named sections, then add design tokens, then fill each section with a targeted replace; never emit a full finished page in one edit, which would hit the `maxOutputTokens` cap mid-generation and truncate); give `from` (and optional `to`) plus `code` to replace a region, or omit `code` to delete it; set `insert: "before"/"after"` to insert `code` relative to `from` (or the document start/end when `from` is omitted). `from`/`to` are order-insensitive (resolved by document position, so reversed endpoints succeed). Edits resolve against the original anchored document, validate atomically, preserve untouched anchors, allocate fresh anchors for inserted/replaced lines, write through to the project store on success, and return concise metadata plus a per-edit result slice (`result.edits[i]` = that edit's own diff/anchors) instead of full HTML. At the SSE boundary a batched edit call (`edits.length >= 2`) fans out into one `tool_call` event per edit (distinct ids, per-edit action/result) so the UI shows one block per edit; a single-edit call stays one block but still derives its displayed action from `edits[0].action` via `defaultToolAction` (there is no call-level action). `client-messages.jsonl` logs every fanned-out `tool_call` event (and the `tool_call_drop` that removes the streaming-start provisional on fan-out); `html_update` still fires once per call (the batch lands as one atomic HTML write).
- `openrouterModel()` (`lib/openrouter-model.ts`) uses Mastra `OpenAICompatibleConfig` with ids shaped as `openrouter/${modelId}` so the synthetic prefix is stripped and the full OpenRouter model id is sent verbatim to `openrouter.ai/api/v1/chat/completions`. OpenRouter is the only provider for all agent LLM traffic (text brain), image generation, and vision OCR. All three roles are selectable per request via `/agent { textModel?, imageModel?, visionModel? }` and default from `config.openrouter.default{Chat,Image,Vision}Model`. Main agent retries are configured through `config.agentRetry`; hidden AI SDK model retries default to `0` so user-visible Mastra retry events own backoff reporting unless overridden by env.
- Streaming tool-call finalization is patched (see root AGENTS + `patches/@mastra__core@1.47.0.patch`): the `tool_call` SSE event for a step now emits at end-of-stream (all argument fragments accumulated), not mid-stream on the AI-SDK's `isParsableJson` early-finalize. This is what stops GLM-5.2 from dropping multi-fragment `edit` arguments as empty `{}`. Incremental `tool-input-delta` UI streaming is unchanged.
- `resolveModelId()` strips an optional `openrouter/` prefix and falls back to the **chat** default only. It is correct for `textModel`, but the `/agent` handler must forward `undefined` for omitted `imageModel`/`visionModel` so `streamLandingAgent`'s `StreamOptions` destructuring defaults apply the role-specific image/vision models. Routing image/vision through `resolveModelId` silently substitutes the chat model (neither image nor vision capable) and 404s at the provider when callers send only `textModel`.
- Vision OCR uses the configured OpenRouter vision model (default `z-ai/glm-5v-turbo`) with the Z.AI `ui_to_artifact` system prompt (the senior-frontend-engineer/code persona: extract structure, spacing scale, hex colors, and typography) sent as the `system` message, while per-call user prompts (attachment framing, screenshot action, scrape OCR) are the `user` message alongside the image parts; image generation uses OpenRouter's `/api/v1/images` endpoint with the configured image model (default `bytedance-seed/seedream-4.5`). OpenRouter cost accounting must use only provider-reported response metadata (`cost`, `total_cost`, `estimated_cost`, or nested cost objects) and must not estimate USD from tokens or image counts. LLM streaming must request raw chunks and capture provider cost metadata when present; when metadata is absent, the OpenRouter cost is `0`. Firecrawl scrape cost is calculated from Firecrawl-reported `creditsUsed` multiplied by `config.firecrawl.creditUsd` (default `0.002`, override with `FIRECRAWL_CREDIT_USD`). Project reads must not normalize legacy zero-cost stats from saved token usage.
- The `screenshot` tool accepts exactly `action`, `selector`, and `viewportSize` (`mobile` | `tablet` | `desktop`). `action` is an optional, agent-supplied string that becomes the vision `user` message alongside the Z.AI `ui_to_artifact` system prompt (e.g. "check hero spacing and CTA contrast"); when omitted it defaults to a generic inspection prompt. The tool emits a project-correlated `screenshot_request` SSE event, waits on the process-local browser screenshot registry, OCRs the returned selector screenshot with the configured OpenRouter vision model, and returns visual QA notes without creating files or auxiliary artifacts. Because `action` is now a schema field, its displayed reason comes from the agent's `action`; the previous selector/viewport-derived action no longer applies. Client capture renders the requested viewport and adds 8px padding around the selected element image. The request callback memoizes timeouts: once a `screenshot_request` times out (no browser client captured it), subsequent `screenshot` calls in the same run fail fast with an actionable reason ("No browser client captured the previous screenshot request...") instead of waiting through the full timeout on every retry.
- `design-skill.ts` is self-contained browser design guidance. Its instructions and references must keep the agent anchored to the project HTML document as the only editable surface and must use `read`/`find` anchors before `edit` ranges.
- The image store is process-memory during a live `generate_image` call; `createProjectHtmlStore`'s sync write path copies those bytes into `.data/projects/<id>/images/` and rewrites their URLs to `/api/projects/:id/images/<file>` as the agent edits the document. The in-memory image store is source only during a run; the project `html.json` file is the source of truth.

## Work Guidance

- For Mastra changes, read installed package docs/types first; Mastra APIs change quickly.
- Keep per-request `Agent` + `HtmlStore` isolation for production requests; the singleton registration exists for Mastra Studio visibility.
- Prefer `zod` schemas at tool boundaries and small pure helpers in `lib/`.
