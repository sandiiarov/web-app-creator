# Client App DOX

## Purpose

- Vite + React app. Manage landing-page projects, prompt landing-page agent, preview each generated single-file HTML.
- `/` lists saved projects; `/projects/new` creates draft, redirects to `/projects/:id`; `/projects/:id` = editor: sandboxed `srcDoc` iframe preview + draggable prompt/conversation panel.

## Ownership

- `src/main.tsx`: `BrowserRouter` + route table (`/`, `/projects/new`, `/projects/:id`), `ThemeProvider` root, `TooltipProvider` root (all tooltips descend from it).
- `src/App.tsx`: `EditorPage` (preview + prompt panel composition), param-driven, keyed by project id. In dev, exported component identity + latest render fn bridged through `import.meta.hot.data`; inline `accept` callback rerenders stable component so App HMR reconciles around existing preview iframe instead of remounting.
- `src/components/projects-page.tsx`: project list + new-project redirect.
- `src/components/`: app-specific UI (error banner, theme provider). `LandingPreview` React component consumed from dedicated `@workspace/landing-preview/react` Fast Refresh boundary (see `packages/landing-preview/AGENTS.md`); preview utilities remain in package barrel. Prompt panel consumed from `@workspace/prompt-panel`; `EditorPage` injects `onAllProjects`, `theme`, `onToggleTheme`, `onLayoutChange`, `onReloadPreview`, sizes preview iframe from reported panel layout (init via `readStoredPanelLayout`).
- `src/hooks/`: streaming hooks.
- `src/lib/`: custom SSE client, landing-agent transport (server URL + SSE wire events; domain model lives in `@workspace/prompt-panel`, re-exported from `landing-agent.ts` as bridge while in-flight files repointed), image URL expansion, project REST API client. Browser screenshot capture + preview iframe/morph helpers live in `@workspace/landing-preview`.
- `components.json`: shadcn project config targeting shared UI code in `packages/ui`.

## Local Contracts

- Server API: `POST /agent` is a fire-and-forget JSON RPC at `VITE_SERVER_URL`/`http://localhost:3001` returning `{ ok, status:'running', turnId }` immediately (overlapping run → rejects); live progress is delivered through `GET /api/projects/:id/events` (per-project SSE subscribe). Prompts may include image attachments (JSON data URLs) or selected-element attachments (selector-only `{ kind:'element', selector }`). Subscribe events: one `state` snapshot `{ html, models, status, turns }` on connect, then the run event stream (`thinking`, `text`, `tool_call`, `html_update`, `retry`, rolling `stats`, `error`, `done`). `retry` renders visible countdown/backoff notice in active prompt panel turn. While applying live `analyze_image` tool events, `use-landing-page.ts` enriches tool args w/ submitted attachment data URLs (held in `pendingAttachmentsRef` for the in-flight send) so expanded diagnostic block previews analyzed images without sending base64 back through server's durable event log. Keep `src/lib/landing-agent.ts` event types aligned w/ `apps/server/src/mastra/route.ts`.
- Each `/agent` send posts a browser-created `turnId`; `send()` appends an optimistic streaming turn, POSTs `sendPrompt`, and the subscribe stream delivers the run's events to that turn (active = `activeTurnIdRef`, the last streaming turn). `sendingRef` = synchronous send lock, so two sends in one render tick cannot start overlapping runs; `sendPrompt` rejection rolls the optimistic turn back to a terminal error. Stopping is graceful: `stop()` calls `POST /api/projects/:id/stop` (`stopProjectAgent`), terminalizes active tools immediately for feedback, keeps panel `isStreaming` true until the subscribe stream delivers terminal `stats`+`error`+`done` (preserves Spend/tokens, blocks sends during drain); does NOT abort the subscribe connection. 8-second `stopSafetyRef` force-finalizes local state if the server never delivers a terminal event. Happy-dom hook tests must cover state-snapshot hydration, send+live-event application, sendPrompt rejection rollback, same-tick send locking, graceful drain/timer cleanup.
- Server owns project's `html.json`/`index.html` — single source of truth. **No client PUT**: on editor mount `useLandingPage` opens the per-project event subscription; the `state` snapshot seeds html/models/turns/status, then live `html_update` events morph the preview iframe. Client expands project image URLs (`expandProjectImageUrls`). A tab reopened mid-run sees live progress (subscribe catch-up uses `replayClientEventsLive` server-side so the in-flight turn renders streaming, not finalized).
- Projects read via `src/lib/projects-api.ts` against `/api/projects*`. Full project reads include persisted `messages`; `use-landing-page.ts` restores them into prompt panel as non-streaming turns on editor mount. Projects list stays metadata-first, but cards may fetch full projects to render sandboxed `srcDoc` iframe previews. Stored HTML uses root-relative project image URLs (`/api/projects/:id/images/<file>`); `expandProjectImageUrls` expands to absolute before rendering editor/card preview iframes.
- Editor preview inits sandboxed `srcDoc` iframe permitting forms, modals, popups, same-origin storage, scripts; applies routine `html_update` changes in-place with same-document DOM morph helper rather than replacing `srcDoc`. Script-changing updates rerun scripts inside existing iframe doc instead of refreshing frame. HMR must preserve live generated iframe document. Keep `EditorPage` on stable `import.meta.hot.data` proxy with inline `import.meta.hot.accept` rerender; keep preview component import on dedicated `@workspace/landing-preview/react` Fast Refresh boundary, not mixed utility barrel. Client dynamically injects `<base href="about:srcdoc">` into preview HTML before initial `srcDoc` loads + morph targets only, so generated `#section` links resolve inside iframe without changing persisted project HTML. Content + message history pulled from server (not generated/persisted in browser). Composer element-selection toggle installs iframe-local capture listeners, resolves hover/click targets with `elementsFromPoint`, blocks generated-page interactions while active, renders animated pointer-events-none dashed accent overlay box around hovered/selected elements, then attaches selected element as selector-only metadata payload `{ kind: 'element', selector }` to prompt. Server captures each selector at mobile/tablet/desktop via Cloudflare Browser Run; client never rasterizes DOM elements. Do not reintroduce full workspace snapshot, client-side message save path, or browser AI SDK tool loop without updating plans + DOX.
- Prompt panel composer action row ordered: element select, image attach, model selector, cumulative project spend. Model selector = single dropdown with 3 separated labeled radio sections — Text, Vision, Image — bound to `LandingModels` selection (`{ text, vision, image }`); every Text selection uses OpenRouter's `:nitro` routing variant, including restored legacy selections; Image + Vision ids unchanged. Selections persist immediately via `PATCH /api/projects/:id { textModel, visionModel, imageModel }`; individual saved message turns still show text model used for that turn. Server persists all 3 selections (`textModel`, `visionModel`, `imageModel`) to project metadata via `PATCH`, survive reload; per-request `/agent` call still sends them so run uses current selection. Spend trigger opens popover with cumulative project cost/category + token totals, updates from rolling stats whenever provider cost/usage becomes known after LLM step or terminal tool event (incl. active turn), replaces old per-turn Spend / tokens blocks. Cost formatting owned by `@workspace/prompt-panel`: popup details display USD with exactly 4 digits after decimal point; icon-free popup trigger shows only dollar amount rounded to 2 digits.
- Prompt panel header action row ordered: All Projects, preview viewport dropdown, refresh preview, Download HTML, panel layout dropdown, settings dropdown, minimize/maximize. Refresh preview button opt-in via `onReloadPreview`; disabled until generated HTML exists (`canDownload`), reloads live preview iframe from current server-owned HTML without server round-trip (consumer bumps `reloadToken` passed to `LandingPreview`). Viewport dropdown owns desktop (default), tablet, mobile; trigger icon reflects current selection (`Monitor`/`Tablet`/`Smartphone`), choices render as plain menu items without checkbox/radio indicators, like layout dropdown. Panel layout dropdown owns left sidebar, right sidebar, floating; trigger uses `PanelsTopLeft` icon. Settings dropdown owns theme toggle. Header dropdowns stay compact: no visible group label headings; layout/theme menu items use shared command shortcut treatment; panel + viewport dropdown triggers themselves have no keyboard shortcut. Command/Control Option A shortcut still navigates to `/`. Panel persists only position/layout + minimized/maximized state in browser localStorage under `landing.promptPanel.position.v1`; viewport selection owned by `EditorPage` as component state, not persisted (defaults to desktop each mount).
- Whole editor layout follows panel's layout + selected preview viewport. `LandingPreview` wrapped in centering container carrying panel-offset margins when docked-and-expanded (`ml-[var(--landing-panel-width)]` when panel on left, `mr-[var(--landing-panel-width)]` when on right, each `w-[calc(100vw-var(--landing-panel-width))]` + `flex justify-center`) and is full-viewport `h-svh w-screen` layer when floating/collapsed. Inside it iframe sized to viewport via `iframeClassName` — desktop fills area, tablet + mobile render at fixed widths + stay centered (with `shrink-0` so exact viewport width preserved) — so panel + preview never overlap, landing page stays fully visible (panel's existing edge border = divider). When tablet/mobile selected, preview area also shows subtle 10px grid (`landing-grid-bg`, defined in `packages/ui` globals) behind centered iframe. `--landing-panel-width` CSS var owned by `@workspace/prompt-panel`, not editor: registered as typed `@property` (declared in `packages/ui` globals, with CSS `initial-value` default so no first-paint flash + can be transitioned later), panel writes runtime value on mount + while resizing. `EditorPage` only consumes `var(--landing-panel-width)` for docked preview offset, holds no width state (no longer imports `PANEL_WIDTH`). Panel horizontally resizable from both edges between `MIN_PANEL_WIDTH` + half viewport (`maxPanelWidth`); width persists alongside position. Preview's dock offset (on `[data-landing-preview-area]`) is CSS-transitioned on dock/undock, disabled under `:has([data-resizing])` so it tracks `--landing-panel-width` exactly during panel resize (rules live in `packages/ui` globals). `EditorPage` derives layout from panel's `onLayoutChange` reports (collapsed panel reports `floating`), inits from `readStoredPanelLayout` to avoid first-paint flash; viewport from local `EditorPage` state. Do not write panel width or viewport widths as class literals in docs — Tailwind v4 auto-scans markdown, would generate utilities from doc text.
- Prompt message markdown renders with Streamdown; code fences must use `@streamdown/code` syntax highlighting with Catppuccin Latte/Mocha configured on code plugin (Streamdown reads `plugins.code.getThemes()` before `shikiTheme` prop) + panel-scoped Streamdown styling so headings, tables, lists, inline code, code blocks match square/sharp UI language. Keep highlighted code line wrappers block-level when line numbers off so multiline fences do not collapse into one visual line. Code-block chrome must stay square/no-radius, hide copy/download controls.
- Prompt diagnostic blocks (`tool_call` + `thinking` parts) render as individual blocks, not grouped clusters. Blocks stay collapsed unless user explicitly expands them; streaming state must not auto-open/auto-close details. State communicated through color treatment instead of status chips; only latest streamed part shows active/loading treatment. Thinking keeps own treatment: brain icon + `Thinking` header, thinking text in body. Every tool call uses same collapsible component shape: header row with icon + tool name, untruncated multiline subheader intent, expanded body containing args + result/error.
- Use `@workspace/ui/...` + `@workspace/prompt-panel` for shared package imports; `#components`, `#hooks`, `#lib` for app-local aliases.
- Do not put secrets in client code; only `VITE_*` variables client-readable.
- Preserve sharp/square visual language unless design direction explicitly changed.

## Work Guidance

- Follow shadcn/Tailwind rules: semantic tokens, `cn()` for conditional classes, `gap-*` over `space-*`, `size-*` for square dimensions, existing UI components before custom markup.
- Keep streaming state changes in hooks; keep prompt UI components mostly presentational.
- Update client event model + server SSE mapping together. Failed tool calls + failed turns must render with destructive/red styling; any still-running tool rows must be terminalized when run errors, completes, or is stopped. Agent-run errors stay inside turn; reserve global error banner for app/preview/load failures.
