# Client App DOX

## Purpose

- Vite + React app that lets a user manage landing-page projects, prompt a landing-page agent, and preview each generated single-file HTML.
- `/` lists saved projects; `/projects/new` creates a draft and redirects to `/projects/:id`; `/projects/:id` is the editor: a sandboxed `srcDoc` iframe preview plus a draggable prompt/conversation panel.

## Ownership

- `src/main.tsx`: `BrowserRouter` + route table (`/`, `/projects/new`, `/projects/:id`), `ThemeProvider` root, and `TooltipProvider` root (all tooltips in the app descend from it).
- `src/App.tsx`: `EditorPage` (preview + prompt panel composition), param-driven and keyed by project id. In development its exported component identity and latest render function are bridged through `import.meta.hot.data`; the inline `accept` callback rerenders the stable component so App HMR reconciles around the existing preview iframe instead of remounting it.
- `src/components/projects-page.tsx`: project list + new-project redirect.
- `src/components/`: app-specific UI (error banner, theme provider). The `LandingPreview` React component is consumed from the dedicated `@workspace/landing-preview/react` Fast Refresh boundary (see `packages/landing-preview/AGENTS.md`); preview utilities remain available from the package barrel. The prompt panel is consumed from `@workspace/prompt-panel`; `EditorPage` injects `onAllProjects`, `theme`, `onToggleTheme`, and `onLayoutChange`, and sizes the preview iframe from the reported panel layout (initialized via `readStoredPanelLayout`).
- `src/hooks/`: streaming hooks.
- `src/lib/`: custom SSE client, landing-agent transport (server URL + SSE wire events; the domain model lives in `@workspace/prompt-panel` and is re-exported from `landing-agent.ts` as a bridge while in-flight files are repointed), image URL expansion, and the project REST API client. Browser screenshot capture and preview iframe/morph helpers live in `@workspace/landing-preview`.
- `components.json`: shadcn project config that targets shared UI code in `packages/ui`.

## Local Contracts

- The server API is custom SSE `POST /agent` at `VITE_SERVER_URL` or `http://localhost:3001`; prompts may include image attachments as JSON data URLs or selected-element attachments with HTML plus screenshot data URLs, and streamed events include `html_update`, `retry`, and `screenshot_request` in addition to text/tool/rolling-stats/error/done events. `retry` events render a visible countdown/backoff notice in the active prompt panel turn. While applying live `analyze_image` tool events, `use-landing-page.ts` enriches the tool args with the submitted attachment data URLs so the expanded diagnostic block can preview the analyzed images without sending base64 back through the server's durable event log. Keep `src/lib/landing-agent.ts` event types aligned with `apps/server/src/mastra/route.ts`.
- Each first-party `/agent` request sends the browser-created `turnId`; the same ID is persisted and hydrated so a rejected stream or transport EOF without custom `done` can poll `GET /api/projects/:id` on a short bounded schedule and replace only that matching local turn once it is terminal. The controller ref is the synchronous send lock as well as the cancellation handle, so two sends in one render tick cannot start overlapping runs. Stopping is graceful: `use-landing-page.ts` `stop()` calls `POST /api/projects/:id/stop` (`stopProjectAgent` in `src/lib/projects-api.ts`), which aborts server generation without closing SSE. The client marks the local turn `stopped`, terminalizes active tools/stops their animations, but keeps panel `isStreaming` true and keeps reading until terminal `stats` + stopped + `done` arrive; this preserves Spend / tokens and blocks sends during drain. It does not abort fetch unless the endpoint fails or the 8-second `stopSafetyRef` fallback fires. `stoppingRef` blocks repeated stops, and the run finalizer clears the timer/refs. Happy-dom hook tests must cover same-tick send locking, graceful drain/stats/timer cleanup, and matching-turn transport reconciliation.
- The server owns the project's `index.html` file — it is the single source of truth. There is **no client PUT**: on editor mount the UI reads `GET /api/projects/:id`, and after successful content-changing `edit` tool results the server streams `html_update` with the current HTML. The client expands project image URLs and morphs the live preview iframe; fallback reloads may still use the server-owned HTML.
- Projects are read via `src/lib/projects-api.ts` against `/api/projects*`. Full project reads include persisted `messages`; `use-landing-page.ts` restores them into the prompt panel as non-streaming turns on editor mount. The projects list stays metadata-first, but cards may fetch full projects to render sandboxed `srcDoc` iframe previews. Stored HTML uses root-relative project image URLs (`/api/projects/:id/images/<file>`); `expandProjectImageUrls` expands them to absolute before rendering editor or card preview iframes.
- The editor preview initializes a sandboxed `srcDoc` iframe that permits forms, modals, popups, same-origin storage, and scripts, then applies routine `html_update` changes in-place with a same-document DOM morph helper rather than replacing `srcDoc`. Script-changing updates rerun scripts inside the existing iframe document instead of refreshing the frame. HMR must preserve the live generated iframe document. Keep `EditorPage` on its stable `import.meta.hot.data` proxy with an inline `import.meta.hot.accept` rerender, and keep the preview component import on the dedicated `@workspace/landing-preview/react` Fast Refresh boundary rather than the mixed utility barrel. The client dynamically injects `<base href="about:srcdoc">` into preview HTML before initial `srcDoc` loads and morph targets only so generated `#section` links resolve inside the iframe without changing persisted project HTML. The content and message history are pulled from the server (they are not generated or persisted in the browser). The composer element-selection toggle installs iframe-local capture listeners, resolves hover/click targets with `elementsFromPoint`, blocks generated-page interactions while active, renders an animated pointer-events-none dashed accent overlay box around hovered/selected elements, then attaches the selected element's `outerHTML` plus padded screenshot to the prompt. For `screenshot_request`, the client pulls the latest project HTML from the server, renders it in a temporary same-origin/no-script offscreen iframe at the requested `mobile`/`tablet`/`desktop` viewport, captures the requested selector with `@zumer/snapdom`, rasterizes it at half its CSS dimensions, adds 8px padding around the output image, and posts the correlated response to `/api/screenshot-responses/:requestId`. Do not reintroduce a full workspace snapshot, client-side message save path, or browser AI SDK tool loop without updating plans and DOX.
- The prompt panel composer action row is ordered as element select, image attach, model selector, then cumulative project spend. The model selector is a single dropdown with three separated, labeled radio sections — Text, Vision, Image — bound to a `LandingModels` selection (`{ text, vision, image }`); every Text selection uses OpenRouter's `:nitro` routing variant, including restored legacy selections, while Image and Vision ids are unchanged. Selections persist immediately via `PATCH /api/projects/:id { textModel, visionModel, imageModel }`; individual saved message turns still show the text model used for that turn. The server persists all three selections (`textModel`, `visionModel`, `imageModel`) to project metadata via `PATCH`, so they survive reload; the per-request `/agent` call still sends them so a run uses the current selection. The spend trigger opens a popover with cumulative project cost/category and token totals, updates from rolling stats whenever provider cost/usage becomes known after an LLM step or terminal tool event (including the active turn), and replaces the old per-turn Spend / tokens blocks. Cost formatting is owned by `@workspace/prompt-panel`: popup details display USD with exactly four digits after the decimal point, while the icon-free popup trigger shows only the dollar amount rounded to two digits.
- The prompt panel header action row is ordered as All Projects, preview viewport dropdown, Download HTML, panel layout dropdown, settings dropdown, then minimize/maximize. The viewport dropdown owns desktop (default), tablet, and mobile choices; its trigger icon reflects the current selection (`Monitor`/`Tablet`/`Smartphone`) and its choices render as plain menu items without checkbox/radio indicators, like the layout dropdown. The panel layout dropdown owns left sidebar, right sidebar, and floating choices; its trigger uses the `PanelsTopLeft` icon. The settings dropdown owns the theme toggle. Header dropdowns stay compact: no visible group label headings; layout/theme menu items use the shared command shortcut treatment; the panel and viewport dropdown triggers themselves have no keyboard shortcut. The Command/Control Option A shortcut still navigates to `/`. The panel persists only position/layout plus minimized/maximized state in browser localStorage under `landing.promptPanel.position.v1`; the viewport selection is owned by `EditorPage` as component state and is not persisted (defaults to desktop each mount).
- The whole editor layout follows the panel's layout and the selected preview viewport. `LandingPreview` is wrapped in a centering container that carries the panel-offset margins when docked-and-expanded (`ml-[var(--landing-panel-width)]` when the panel is on the left, `mr-[var(--landing-panel-width)]` when on the right, each `w-[calc(100vw-var(--landing-panel-width))]` plus `flex justify-center`) and is a full-viewport `h-svh w-screen` layer when floating or collapsed. Inside it the iframe is sized to the viewport via `iframeClassName` — desktop fills the area, while tablet and mobile render at fixed widths and stay centered (with `shrink-0` so the exact viewport width is preserved) — so the panel and preview never overlap and the landing page stays fully visible (the panel's existing edge border is the divider). When tablet or mobile is selected, the preview area also shows a subtle 10px grid (`landing-grid-bg`, defined in `packages/ui` globals) behind the centered iframe. The `--landing-panel-width` CSS var is owned by `@workspace/prompt-panel`, not the editor: it is registered as a typed `@property` (declared in `packages/ui` globals, with a CSS `initial-value` default so there is no first-paint flash and it can be transitioned later) and the panel writes its runtime value on mount and while resizing. `EditorPage` only consumes `var(--landing-panel-width)` for the docked preview offset and holds no width state (it no longer imports `PANEL_WIDTH`). The panel is horizontally resizable from both edges between `MIN_PANEL_WIDTH` and half the viewport (`maxPanelWidth`), and the width persists alongside position. The preview's dock offset (on `[data-landing-preview-area]`) is CSS-transitioned on dock/undock, but disabled under `:has([data-resizing])` so it tracks `--landing-panel-width` exactly during a panel resize (rules live in `packages/ui` globals). `EditorPage` derives the layout from the panel's `onLayoutChange` reports (a collapsed panel reports `floating`) and initializes from `readStoredPanelLayout` to avoid a first-paint flash; the viewport comes from local `EditorPage` state. Do not write the panel width or viewport widths as class literals in docs — Tailwind v4 auto-scans markdown and would generate utilities from doc text.
- Prompt message markdown renders with Streamdown; code fences must use `@streamdown/code` syntax highlighting with Catppuccin Latte/Mocha configured on the code plugin (Streamdown reads `plugins.code.getThemes()` before the `shikiTheme` prop) and panel-scoped Streamdown styling so headings, tables, lists, inline code, and code blocks match the square/sharp UI language. Keep highlighted code line wrappers block-level when line numbers are off so multiline fences do not collapse into one visual line. Code-block chrome must stay square/no-radius and hide copy/download controls.
- Prompt diagnostic blocks (`tool_call` and `thinking` parts) render as individual blocks, not grouped clusters. Blocks stay collapsed unless the user explicitly expands them; streaming state must not auto-open or auto-close details. State is communicated through color treatment instead of status chips, and only the latest streamed part shows active/loading treatment. Thinking keeps its own treatment: brain icon plus `Thinking` header, with thinking text in the body. Every tool call uses the same collapsible component shape: header row with icon plus tool name, untruncated multiline subheader intent, and expanded body containing args plus result/error.
- Use `@workspace/ui/...` and `@workspace/prompt-panel` for shared package imports, and `#components`, `#hooks`, `#lib` for app-local aliases.
- Do not put secrets in client code; only `VITE_*` variables are client-readable.
- Preserve the sharp/square visual language unless the design direction is explicitly changed.

## Work Guidance

- Follow shadcn/Tailwind rules: semantic tokens, `cn()` for conditional classes, `gap-*` over `space-*`, `size-*` for square dimensions, and existing UI components before custom markup.
- Keep streaming state changes in hooks and keep prompt UI components mostly presentational.
- Update the client event model and server SSE mapping together. Failed tool calls and failed turns must render with destructive/red styling and any still-running tool rows must be terminalized when a run errors, completes, or is stopped. Agent-run errors stay inside the turn; reserve the global error banner for app/preview/load failures.
