# Create

Use create to build a new landing page or add a genuinely new section to the existing project HTML. The root skill owns required reference loading; this file owns the creation workflow after that gate is complete.

## Start from the actual project

Read the current HTML first. A new project already contains a placeholder document. Build into that document incrementally rather than creating another file or emitting one oversized replacement.

Extract the prompt invariants before composing:

- Exact name and category
- Audience and immediate pressure
- Offer and primary action
- Concrete domain artifact
- Credible proof
- Required content, assets, constraints, and exclusions
- Visual or verbal drift to refuse

Ask one question only when a missing answer changes the product or target. Decide normal style details yourself.

## Define the page premise

Write one internal sentence describing the visual world and one sentence describing the visitor's path. The page premise should explain why this layout, type, color, imagery, and proof belong to this brief.

Choose a proof object visitors can inspect: a real workflow, record, transformation, comparison, environment, product view, result, or physical subject. An abstract panel beside a headline is not proof by itself.

## Build in layers

### 1. Story and structure

Establish semantic landmarks and a purposeful section order. A typical page may need arrival, proof, explanation, objection handling, trust, and action, but do not add formulaic sections merely to fill a template.

Use real copy or realistic supplied content. Preserve exact claims and names. Do not leave placeholder text, dead links, or invented testimonials.

### 2. Foundation

Create a small set of CSS custom properties for color, type, spacing, edges, depth, and motion actually used by the page. Avoid a token inventory with no consumers.

Set the page shell, content measure, grid, and responsive order before adding decorative effects.

### 3. First viewport and proof

Make category, voice, proof, and primary action legible immediately. The hero and proof object should share one composition rather than look like unrelated halves.

Use scraped brand assets when supplied and suitable. Generate imagery only after the root reference gate and only when it strengthens the chosen premise. Physical subjects generally need specific imagery; generic software can be type-, diagram-, or product-artifact-led.

### 4. Sections and controls

Build one coherent section at a time. Vary rhythm according to content priority rather than alternating identical blocks. Buttons, links, forms, menus, accordions, and demos receive only the states they can actually enter.

Loading, error, empty, success, disabled, and overlay states are required only when real asynchronous or interactive elements need them. Do not invent application behavior for a static page.

### 5. Responsive behavior

Recompose the same story for mobile, tablet, and desktop. Preserve the primary action, proof, and content order; do not merely squeeze the desktop layout. Use content-driven CSS breakpoints where needed.

### 6. Motion and finish

Add motion only when it guides attention, explains state, or reinforces the brand premise. Implement reduced-motion behavior. A page can intentionally use no expressive motion.

## Verification

After substantial edits, inspect desktop and mobile screenshots and tablet when composition changes there. Check source for semantics, links, focus styles, media queries, reduced motion, and hidden states. Fix observed clipping, weak hierarchy, generic reflexes, and unfinished content before responding.

## Done when

- The page is built into the real project HTML through incremental edits
- Name, category, proof, and action are specific to the prompt
- The first viewport commits to a recognizable lane
- Sections form a coherent decision, learning, exploration, or comparison path
- Real controls have appropriate states without invented product scope
- Relevant screenshots and source checks support every completion claim
