# Redesign

Use redesign when the user wants the same landing-page product and message to inhabit a genuinely different visual world. Redesign changes the system end to end; it is not a palette swap, hero makeover, or collection of louder effects.

This file governs how to transform the page while preserving its product, message, evidence, and working landing-page behavior.

## Preserve the product

Keep unless the user explicitly changes them:

- Exact name, offer, audience, and claims
- Content priority and primary conversion goal
- Valid links, forms, scripts, and product behavior
- Credible proof and supplied assets
- Accessibility and performance expectations

Do not silently rewrite product scope to make a visual concept easier.

## Diagnose the old premise

Before choosing a new direction, identify:

- The current spatial premise and reading path
- The visual lane, type voice, color mood, edge language, depth, and motion
- What works and should survive
- What makes the page generic, incoherent, or wrong for the brief
- Which parts are content problems rather than visual-system problems

A redesign must change the premise, not merely its values.

## Name the new direction

Choose a concise physical or cultural description that can guide every system: for example, field notebook, gallery placard, technical instrument, theater poster, product workshop, or tactile catalog. Name what the direction is not so trend defaults cannot creep back in.

The direction should determine:

- Composition and section rhythm
- First-viewport proof and imagery
- Type family, scale, and density
- Color roles and contrast
- Buttons, frames, radius, and depth
- Responsive recomposition
- Interaction and motion personality

Different does not mean louder. A redesign can become quieter, more editorial, more rigorous, more tactile, more direct, or more image-led. Avoid choosing hard borders, offset shadows, neon gradients, oversized pills, or sticker shapes merely because they make change obvious.

## Rebuild coherently

Establish the new tokens and page shell, then carry the direction through every requested section. Replace old system values deliberately; do not leave old shadows, radii, spacing, or motion in isolated components.

Apply the new language to states that real controls can enter. Preserve source semantics and behavior while changing presentation.

## Verify distance and continuity

Inspect full-page desktop and mobile screenshots and tablet when its composition differs. The old and new versions should read as different visual products at thumbnail distance, while the same name, offer, proof, and action remain recognizable.

Inspect source for links, forms, focus styles, media queries, reduced motion, and any hidden states. If an interaction cannot be exercised, describe it as source-inspected.

## Done when

- Composition visibly changed before color or font details are considered
- Every requested section belongs to one new visual world
- The page remains truthful to the original product promise and content
- Real controls and responsive states use the new language
- No accessibility or working-behavior regression is evident
- The result feels authored for this brief rather than assembled from a fashionable lane
